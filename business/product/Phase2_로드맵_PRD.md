# Phase 2 로드맵 / PRD 초안

기준일: 2026-03-19  
상태: 검토 완료, 구현 전 우선순위 확정 문서

## 1. 목적

- Phase 1에서 완성한 커뮤니티/운영/보안 기반 위에 어떤 기능을 어떤 순서로 올릴지 결정한다.
- `마켓`, `케어`, `결제`, `공동구매`, `카카오맵` 같은 Phase 2 후보를 한 번에 열지 않고, 운영 리스크와 데이터 밀도를 기준으로 단계화한다.
- 다음 구현 사이클이 바로 착수할 수 있도록 범위, 선행조건, 성공지표, 보류 조건을 명시한다.

## 2. 현재 출발점

- Community core, search, moderation, alert, structured input, upload/media, public profile, auth hardening은 운영 가능한 수준으로 정리됐다.
- 품종 개인화와 광고/운영 지표는 이미 `/feed`, `/admin/personalization`, `/admin/policies`, `/admin/breeds`까지 들어가 있다.
- 운영 측면에서는 `/api/health`, 신고 큐, 인증 감사 로그, 병원 후기 의심 큐, 모더레이션 로그가 준비돼 있다.
- 아직 없는 것:
  - 거래 상태 머신
  - 실거래/정산/환불/분쟁 시스템
  - 지도 기반 탐색/반경 검색
  - 케어/시터/병원 예약성 워크플로우
  - search quality tuning을 위한 운영 대시보드 기반 루프의 실사용 데이터

## 3. Phase 2 우선순위 결론

### Phase 2A: 검색/개인화/운영 증폭

목적
- 이미 있는 커뮤니티를 더 잘 찾게 만들고, 더 잘 운영하게 만든다.

포함 범위
- 초성/오타/띄어쓰기 대응 검색
- 검색 실패어/저성과 검색어 운영 루프
- `/admin/ops` 기반의 일일 운영 루틴
- 품종 개인화 성과 기반 추천/광고 튜닝

왜 먼저 하는가
- 거래/결제보다 리스크가 낮고, 지금 있는 사용자 행동 데이터를 바로 레버리지할 수 있다.
- Community density를 먼저 높여야 마켓/케어 전환도 의미가 생긴다.

성공지표
- 검색 zero-result 비율 감소
- 검색 CTR 상승
- personalized feed CTR/ad CTR 유지 또는 상승
- 신고 적체/로그인 실패 급증 없는 상태 유지

### Phase 2B: 마켓 상태 머신

목적
- 현재 `MARKET_LISTING`을 “글” 수준에서 “상태가 있는 거래” 수준으로 올린다.

포함 범위
- 거래 상태: `OPEN -> RESERVED -> COMPLETED/CANCELLED`
- 판매/나눔/대여 구분
- 판매자/구매자 역할 및 상태 변경 로그
- 문의/협의 동선과 신고/제재 연동

보류 범위
- 결제/정산/수수료
- 에스크로
- 배송 연동

선행조건
- 운영 정책: [마켓_운영규칙.md](/Users/alex/project/townpet/business/policies/%EB%A7%88%EC%BC%93_%EC%9A%B4%EC%98%81%EA%B7%9C%EC%B9%99.md) 구체화
- moderation action log 기반 분쟁 추적
- 신규 계정/다계정 abuse 대응이 현재보다 더 강해질 것

### Phase 2C: 케어/구인구직/돌봄 요청

목적
- “지금 필요한 도움”을 구조화해서 지역 단위로 매칭한다.

포함 범위
- 요청 템플릿(날짜, 지역, 대상 반려동물, 요청 유형)
- 지원/문의 상태
- 운영자 검토 및 신고/차단 연동

보류 범위
- 예약/결제 보증
- 외부 계약/보험 자동화

선행조건
- 지역 검색과 구조화 필드가 충분히 안정적일 것
- 마켓과 마찬가지로 상태 머신/감사 로그가 준비될 것

### Phase 2D: 카카오맵/지도 탐색

목적
- 병원/산책/장소 후기의 탐색 비용을 줄인다.

포함 범위
- 장소/병원/산책코스 지도 핀
- 지역/반경 기반 검색
- 상세 페이지에서 지도 컨텍스트 표시

보류 범위
- 실시간 길찾기/예약
- 장소 데이터 대량 동기화

선행조건
- 장소/병원명/지역명 canonicalization 안정화
- search/structured field 운영 데이터가 쌓일 것

### Phase 2E: 결제/정산

결론
- 가장 나중에 한다.

이유
- 결제는 기능이 아니라 운영 시스템이다.
- 환불, 영수증, 분쟁, 정산, 수수료, 회계, 법적 고지가 모두 선행돼야 한다.

착수 조건
- 마켓/공동구매/케어에서 실제 거래 의도가 충분히 검증될 것
- dispute/증빙/상태머신 로그가 누적될 것
- 운영자가 결제 없는 상태에서도 충분히 분쟁을 다뤄본 뒤일 것

## 6. Go / No-Go 기준

Phase 2 기능은 아래 조건을 만족할 때만 동시에 2개 이상 병행한다.

- 14일 연속 `/api/health` 주요 체크 정상
- 신고 적체와 인증 실패율이 안정적
- zero-result search와 cache bypass 같은 운영 경고가 관리 가능한 수준
- 운영자 수동 루틴으로도 대응 가능한 규모
