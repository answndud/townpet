# 품종 개인화 / 광고 / 커뮤니티 실행계획

기준일: 2026-04-24  
상태: 구현 전 우선순위 확정

## 1. 결론

첫 구현 후보는 `개인화/광고 운영 튜닝 루프`로 고정한다.

이미 프로필 품종 신호, `UserAudienceSegment`, `/feed` 맞춤 추천, 품종 라운지, `/api/feed/personalization`, `/admin/personalization`, `/admin/policies`, `/admin/breeds`가 존재한다. 따라서 다음 사이클은 새 도메인 기능을 크게 열기보다 현재 개인화 표면을 운영자가 판단하고 조정할 수 있는 작은 루프로 묶는다.

## 2. 이번 결정에서 제외하는 범위

- 결제, 정산, 환불, 에스크로
- 거래 상태 머신 확장
- 돌봄/구인구직 요청 플로우
- 카카오맵/지도 탐색
- 실제 광고 계약, 과금, 파트너 CRM
- 의료/질병 기반 타겟팅

## 3. 대상 화면과 API

- Public/user surface
  - `/feed`
  - `/lounges/breeds/[breedCode]`
  - `/profile`
  - `/users/[id]`
- Admin surface
  - `/admin/personalization`
  - `/admin/policies`
  - `/admin/breeds`
  - `/admin/ops`
- API/service surface
  - `GET /api/profile/audience-segments`
  - `POST /api/feed/personalization`
  - personalized feed query path
  - breed catalog query/service path
  - policy setting path for feed personalization

## 4. Cycle A 목표

운영자가 최근 개인화 성과를 보고 다음 행동을 결정할 수 있게 한다.

- personalized feed CTR/ad CTR의 하락, 무반응, 특정 audience 쏠림을 발견한다.
- 튜닝 가능한 정책값과 실제 결과 지표를 같은 운영 루틴 안에서 해석한다.
- 광고는 커뮤니티 랭킹을 사지 못한다는 원칙을 문서와 UI에서 분리한다.
- 품종 라운지의 공동구매/거래성 글은 기존 고위험 write 정책을 재사용한다.

## 5. 세부 작업 순서

### A1. 운영 판단 기준 문서화

- 목표: `/admin/personalization`에서 어떤 수치를 보고 어떤 조치를 할지 고정한다.
- 수정 범위: product/operations 문서, 필요 시 관리자 화면 copy.
- 기준 문서: `business/operations/개인화_운영_판단_기준.md`
- 완료 기준:
  - CTR, ad CTR, audience concentration, zero-data 상태의 운영 판정 기준이 문서화된다.
  - 정책값 변경 전후 비교 기간은 기본 14일, 긴급 rollback 7일, 저트래픽/월간 리뷰 30일로 선택한다.
  - 광고 성과를 이유로 커뮤니티 추천 랭킹을 직접 구매하게 만들지 않는다는 Go/No-Go 기준이 문서화된다.

### A2. 관리자 진단 UX 보강

- 목표: 운영자가 빈 데이터, 저성과, 편향 위험을 바로 구분하게 한다.
- 수정 범위: `/admin/personalization` 및 관련 presenter/helper/test.
- 완료 기준:
  - 데이터 없음, 낮은 CTR, 특정 audience 쏠림의 상태 문구가 분리된다.
  - 다음 행동이 `/admin/policies`, `/admin/breeds`, `/admin/ops` 중 어디인지 명확하다.
  - 모바일 폭에서도 KPI 카드와 테이블이 읽힌다.

### A3. 광고/추천 정책 분리 증거

- 목표: 광고 노출이 추천 랭킹을 직접 구매하지 않는다는 원칙을 회귀 테스트로 고정한다.
- 수정 범위: personalization policy helper/service tests.
- 완료 기준:
  - 광고 클릭/노출 신호는 정책 cap 안에서만 약한 engagement 신호로 반영된다.
  - sponsored/partner 표면은 UI 라벨과 frequency cap을 갖는 별도 후보로만 다룬다.

### A4. 품종 라운지 고위험 write gate 점검

- 목표: 품종 라운지 공동구매/거래성 글이 기존 신규유저 제한, 링크/연락처 제한, 신고/자동숨김 정책을 우회하지 않게 한다.
- 수정 범위: groupbuy route/action/service/test, 정책 문서.
- 완료 기준:
  - 품종 라운지 작성 경로가 기존 post write policy와 guest step-up 정책을 재사용한다.
  - 신고 정책은 일반 post와 동일하게 audit trail을 남긴다.

## 6. 정책 게이트

- Local/Global 경계: 품종 라운지는 `GLOBAL` 지식/경험 표면이며, 지역 거래나 돌봄 요청으로 확장하지 않는다.
- 광고: 라벨 필수, frequency cap 필수, 건강 민감정보 기반 타겟팅 금지.
- 의료/건강: 진단/처방 톤 금지, 경험 공유와 정보 제공 톤만 허용.
- 공동구매/거래성 글: 신규유저 제한, 링크/연락처 제한, 신고 자동숨김, 관리자 감사 로그 필수.
- 개인정보: audience key는 품종/체급/생애단계/관심 태그 수준으로만 쓰고 질병/민감정보는 저장하거나 타겟팅하지 않는다.
