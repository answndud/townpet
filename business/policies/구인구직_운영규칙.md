# 04. 돌봄/구인구직 정책(초안)

## 1) 허용 범위
- 산책/급식/방문 돌봄/단기 펫시팅/심부름
- 전문 서비스(훈련/미용)는 자격/경력 표기를 권장

## 2) 금지
- 불법/무허가 행위
- 과도한 개인정보 요구(주민번호 등)
- 사기 유도(선입금 요구 등)

## 3) 공고/요청 작성 제한(초안)
- 신규 계정은 일정 기간 동안 돌봄 요청/공고 등록 제한
- 긴급 요청은 추가 조건 필요(활동 점수/인증)

## 4) 분쟁 대응
- 플랫폼은 중개자이며, 분쟁 처리 정책(증빙/대화 기록)을 명시한다.
- 결제 도입 전에는 “가이드/템플릿”으로 분쟁을 최소화한다.

## 5) Care Request Templates preflight 결정

현재 코드에는 돌봄/구인구직 전용 `PostType`이나 구조화 relation이 없다. 마켓처럼 기존 `Post`를 운영 표면으로 쓰고, 돌봄 요청 필드는 별도 relation으로 붙인다. 이렇게 하면 feed/search/detail/comment/report/auto-hide 경로를 재사용하면서도 돌봄 상태 머신을 분리해 확장할 수 있다.

### M1. 구조화 돌봄 요청 생성/조회
- 상태: 2026-04-26 구현 완료
- 모델 방향: `PostType.CARE_REQUEST` + `CareRequest` relation
- 기본 범위: `LOCAL` 전용, 동네/지역 맥락 필수
- 입력: `careType`, `startsAt`, `endsAt`, `locationNote`, `petNote`, `requirements`, `rewardAmount`, `isUrgent`
- 기본 상태: `OPEN`
- 조회: feed/detail에서 돌봄 유형, 일정, 지역 힌트, 보상 여부, 긴급 여부를 표시
- 정책: 게스트 작성 차단, 신규 유저 제한, 연락처/외부 링크 제한, 금칙어 검사는 기존 post write policy를 재사용하고 구조화 필드도 moderation text에 포함한다.
- 완료 기준: validation/service/query/UI 테스트가 존재하고, 돌봄 요청이 일반 자유글로만 저장되지 않는다.

### M2. 상태 전환/지원 흐름
- 상태: M1 이후 보류
- 상태 전환 후보: `OPEN -> MATCHED -> IN_PROGRESS -> COMPLETED/CANCELLED`
- 작성자: 요청 취소 가능
- 운영자: 모든 상태에서 수동 취소/복구 가능
- 지원자/수행자: M2에서는 별도 `CareApplication` 또는 inquiry 모델을 검토한다.
- 감사 로그: 상태 변경은 `ModerationActionLog`에 `CARE_STATUS_CHANGED` 같은 action을 추가해 남긴다.
- 분쟁 연결: 신고 사유 `FRAUD`, `PRIVACY`, `EMERGENCY`는 관리자 큐에서 돌봄 상태와 함께 보이게 한다.
- 제외: 결제, 예약 보증, 보험, 외부 계약 자동화, 실시간 위치 추적
