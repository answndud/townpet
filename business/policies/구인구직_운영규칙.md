# 04. 돌봄/구인구직 정책(초안)

## 1) 허용 범위
- 산책/급식/방문 돌봄/단기 펫시팅/심부름
- 전문 서비스(훈련/미용)는 자격/경력 표기를 권장

## 2) 금지
- 불법/무허가 행위
- 과도한 개인정보 요구(주민번호 등)
- 사기 유도(선입금 요구 등)

## 3) 공고/요청 작성 제한(초안)
- 신규 계정은 일정 기간 동안 돌봄 요청/공고 등록 제한
- 긴급 요청은 추가 조건 필요(활동 점수/인증)

## 4) 분쟁 대응
- 플랫폼은 중개자이며, 분쟁 처리 정책(증빙/대화 기록)을 명시한다.
- 결제 도입 전에는 “가이드/템플릿”으로 분쟁을 최소화한다.

## 5) Care Request Templates preflight 결정

현재 코드에는 돌봄/구인구직 전용 `PostType`이나 구조화 relation이 없다. 마켓처럼 기존 `Post`를 운영 표면으로 쓰고, 돌봄 요청 필드는 별도 relation으로 붙인다. 이렇게 하면 feed/search/detail/comment/report/auto-hide 경로를 재사용하면서도 돌봄 상태 머신을 분리해 확장할 수 있다.

### M1. 구조화 돌봄 요청 생성/조회
- 상태: 2026-04-26 구현 완료
- 모델 방향: `PostType.CARE_REQUEST` + `CareRequest` relation
- 기본 범위: `LOCAL` 전용, 동네/지역 맥락 필수
- 입력: `careType`, `startsAt`, `endsAt`, `locationNote`, `petNote`, `requirements`, `rewardAmount`, `isUrgent`
- 기본 상태: `OPEN`
- 조회: feed/detail에서 돌봄 유형, 일정, 지역 힌트, 보상 여부, 긴급 여부를 표시
- 정책: 게스트 작성 차단, 신규 유저 제한, 연락처/외부 링크 제한, 금칙어 검사는 기존 post write policy를 재사용하고 구조화 필드도 moderation text에 포함한다.
- 완료 기준: validation/service/query/UI 테스트가 존재하고, 돌봄 요청이 일반 자유글로만 저장되지 않는다.

### M2. 상태 전환/지원 흐름
- 상태: 2026-04-26 구현 완료
- M2 범위: 요청 상태 전환 액션만 구현한다.
- 상태 전환:
  - 작성자: `OPEN -> CANCELLED`
  - 관리자: 모든 상태에서 필요한 수동 복구/취소 가능
  - 일반 사용자/게스트: 전환 불가
- 지원자/수행자: `CareApplication` 또는 inquiry 모델은 M3로 분리한다.
- 감사 로그: 상태 변경은 `ModerationActionLog`에 `CARE_STATUS_CHANGED` action을 추가해 남긴다.
- 분쟁 연결: 신고 사유 `FRAUD`, `PRIVACY`, `EMERGENCY`는 관리자 큐에서 돌봄 상태와 함께 보이게 한다.
- 제외: 지원자 매칭, 수행자 배정, 결제, 예약 보증, 보험, 외부 계약 자동화, 실시간 위치 추적

### M3. 지원/문의 흐름
- 상태: 2026-04-26 구현 완료
- 모델 방향: 댓글 확장이 아니라 별도 `CareApplication` relation을 둔다.
- 이유: 댓글은 공개 토론/신고/알림 경로이고, 지원은 신청자 상태, 작성자 승인/거절, 취소/노쇼 기록, 개인정보 비노출 정책이 필요한 별도 워크플로우다.
- 입력: `message`만 허용하고 연락처/외부 링크는 기존 comment/post 정책처럼 차단한다.
- 상태: `PENDING`, `ACCEPTED`, `DECLINED`, `CANCELLED`
- 권한:
  - 지원자: 로그인 사용자만 가능, 작성자 본인/게스트/제재 유저/차단 관계는 불가
  - 지원자: 본인 `PENDING` 지원만 취소 가능
  - 작성자: 본인 돌봄 요청의 `PENDING` 지원을 승인/거절 가능
  - 운영자: 수동 거절/복구 범위는 M3 구현 중 최소한으로 제한
- 매칭: 승인 시 `CareRequest.status`를 `MATCHED`로 바꾸고, 이미 매칭된 요청에는 추가 승인을 막는다.
- 알림: M3에서는 지원 생성과 승인/거절 알림만 추가하고, 수행 체크리스트/후기/노쇼 알림은 이후로 분리한다.
- 제외: 공개 댓글 기반 지원, 결제, 예약 보증, 보험, 외부 계약 자동화, 실시간 위치 추적

### M4. 수행 상태/분쟁 preflight 결정
- 상태: 2026-04-26 구현 완료
- 판단: M4는 체크리스트 전체가 아니라 매칭 이후 최소 상태 전환과 분쟁 신호 기록으로 제한한다.
- 바로 구현:
  - 상태 전환: `MATCHED -> IN_PROGRESS -> COMPLETED`
  - 작성자/수락 지원자: 매칭된 요청의 시작/완료 전환 가능
  - 작성자: `MATCHED` 상태에서 요청 취소 가능
  - 운영자: 모든 돌봄 상태를 수동 override 가능
  - 감사 로그: 기존 `CARE_STATUS_CHANGED`에 actor scope, previous/next status, accepted application id를 metadata로 기록
  - 알림: 시작/완료/취소 상태 변경 알림
- 분쟁 연결:
  - `FRAUD`, `PRIVACY`, `EMERGENCY` 신고는 기존 Post 신고 큐를 사용한다.
  - M4에서는 별도 dispute 테이블을 만들지 않고, 관리자 상세에서 돌봄 상태와 매칭 지원자를 함께 확인하는 방향으로 둔다.
  - 운영자 조치는 `ModerationActionLog`와 `ReportAudit`를 소스 오브 트루스로 유지한다.
- 보류:
  - 세부 수행 체크리스트, 체크인/체크아웃 사진, 노쇼 판정, 상호 후기/평점, 별도 분쟁 테이블, 결제/예약 보증/보험/정산, 실시간 위치 추적
- 이유:
  - 결제/보험 없는 상태에서 세부 수행 증빙을 먼저 만들면 책임 범위가 커진다.
  - 현재 단계는 커뮤니티 매칭 신뢰를 올리는 최소 상태 기록과 운영자 개입 가능성만 확보한다.
- 구현 결과:
  - `CARE_STATUS_CHANGED` 알림을 추가했다.
  - 작성자/수락 지원자는 시작/완료 전환을 할 수 있고, 작성자는 매칭 상태에서 취소할 수 있다.
  - 운영자는 상태 override를 수행할 수 있고, 모든 변경은 기존 audit 경로로 기록한다.

### M5. 후기/노쇼/증빙 preflight 결정
- 상태: 2026-04-26 구현 완료
- 판단: M5는 공개 후기/평점이 아니라 비공개 완료 피드백과 이슈 신호 수집으로 제한한다.
- 바로 구현:
  - 모델 방향: `CareCompletionFeedback`
  - 대상: `COMPLETED` 상태의 돌봄 요청과 수락된 `CareApplication`
  - 작성자: 요청 작성자와 수락 지원자만 각 1회 작성
  - 입력: `outcome`, `comment`, `wouldRepeat`, `issueType`
  - 공개 노출: 기본 비공개, 관리자와 당사자만 조회
  - 운영 연결: `issueType`이 `NO_SHOW`, `SAFETY`, `PAYMENT_OR_FRAUD`, `PRIVACY`이면 관리자 확인 큐/신고 유도
  - 감사/알림: 피드백 작성 알림은 상대방에게 보내지 않고, 운영 이슈만 관리자 검토 대상으로 남긴다.
- 보류:
  - 공개 별점/프로필 점수, 자동 평판 랭킹, 체크인/체크아웃 사진, 실시간 위치, 자동 노쇼 판정, 자동 패널티, 결제/보험/정산 연동
- 이유:
  - 공개 평점은 보복/명예훼손/조작 리스크가 높고, 초기 운영 규모에서는 관리자 확인 없는 자동 노출이 위험하다.
  - 결제 없는 단계에서는 증빙 첨부보다 반복 매칭 신뢰와 운영 이슈 신호를 먼저 쌓는 것이 안전하다.
- 구현 결과:
  - `CareCompletionFeedback`, `CareFeedbackOutcome`, `CareFeedbackIssueType`, `CareFeedbackAuthorRole`을 추가했다.
  - 작성자/수락 지원자만 `COMPLETED` 요청에 각 1회 비공개 피드백을 남길 수 있다.
  - `ISSUE` outcome은 `NONE`이 아닌 issue type을 요구한다.
  - 신규 유저 연락처 제한과 금칙어 검사는 comment 입력에도 적용한다.
  - detail API/UI에서는 당사자와 관리자에게만 피드백을 노출한다.
  - 운영자 이슈 큐/필터는 M6에서 범위를 확정한다.

### M6. 운영 신호 큐 preflight 결정
- 상태: 2026-04-26 구현 완료
- 판단: 돌봄 완료 피드백 이슈는 기존 신고 큐에 섞지 않고 별도 관리자 큐로 본다.
- 이유:
  - 신고 큐는 사용자가 명시적으로 제기한 `Report`와 `ReportStatus/ReportAudit` 처리 모델을 전제로 한다.
  - `CareCompletionFeedback`은 당사자 비공개 후기 성격의 운영 신호라 승인/기각/자동숨김과 바로 연결하면 과잉 제재 위험이 있다.
- 바로 구현:
  - 관리자 전용 `/admin/care-feedbacks` 큐
  - `issueType != NONE` 기본 조회, issueType/outcome 필터
  - 관련 돌봄 게시글 상세 링크와 작성자/수락 지원자 맥락 표시
  - `/admin/ops`에 미확인 이슈 신호 수 요약
  - 공개 피드/검색/프로필 비노출 유지
- 보류:
  - 자동 제재, 자동 숨김, 공개 평판 점수, 증빙 업로드, 결제/보험 분쟁 처리, 별도 dispute table
- 구현 결과:
  - `issueType != NONE` 완료 피드백을 조회하는 관리자 query와 `/admin/care-feedbacks` 큐를 추가했다.
  - issueType/outcome 필터, 관련 돌봄 게시글 상세 링크, 요청자/지원자 맥락을 표시한다.
  - `/admin/ops` 운영 큐 요약에 돌봄 이슈 신호 수를 추가했다.
  - 관리자 내비게이션에 돌봄 이슈 신호 항목을 추가했다.
  - 기존 신고 큐, 공개 피드, 검색, 프로필에는 노출하지 않는다.
