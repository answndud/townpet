## ADR-SEC-001: Keep security planning in a dedicated track
- Date: 2026-02-25
- Status: accepted
- Context
- Security work was being mixed into the main plan/progress timeline, making follow-up and audits harder.
- Decision
- Maintain a dedicated security track under `business/security/` while preserving root `PLAN.md`/`PROGRESS.md` as global single source of truth.
- Consequences
- Security work gains focused lifecycle tracking and clearer release readiness checks.
- Slight process overhead is introduced to keep root docs synchronized with short link/status updates.

## ADR-SEC-003: Split public and internal health diagnostics
- Date: 2026-02-25
- Status: accepted
- Context
- `/api/health` was returning environment missing keys and raw database/rate-limit diagnostic details in public responses.
- Decision
- Keep `/api/health` available for public availability checks, but gate detailed diagnostics behind an internal token (`HEALTH_INTERNAL_TOKEN`) passed via `x-health-token` or Bearer token.
- Consequences
- External monitors can continue to check `ok/status` without receiving sensitive details.
- Operators can still access deep diagnostics when authenticated with the internal token.
- 2026-04-04 follow-up
- `HEALTH_INTERNAL_TOKEN`이 비어 있는 non-production에서도 무토큰 detailed diagnostics는 localhost 요청으로만 제한한다.
- shared/public dev host에서는 토큰 없는 상세 health/CSP stats 접근을 허용하지 않는다.

## ADR-SEC-004: Trusted-proxy-aware client IP resolution
- Date: 2026-02-25
- Status: accepted
- Context
- Relying on the first `x-forwarded-for` IP can be spoof-prone when proxy assumptions are unclear.
- Decision
- Resolve client IP using trusted proxy hops:
  - production fallback: `TRUSTED_PROXY_HOPS=1`
  - development/test fallback: `0`
  - explicit override via `TRUSTED_PROXY_HOPS` (bounded)
- Consequences
- Rate-limit and abuse controls are less sensitive to naive header spoofing.
- Infra teams must keep hop count aligned with actual proxy topology.

## ADR-SEC-005: CSP hardening via phased rollout
- Date: 2026-02-25
- Status: accepted
- Context
- Immediate strict CSP enforcement can break existing inline script assumptions.
- Decision
- Roll out CSP hardening in phases:
  - Phase 1: keep current enforce CSP and add strict `content-security-policy-report-only`
  - Phase 2: enable strict enforce via `CSP_ENFORCE_STRICT=1` after violation review
- Consequences
- XSS hardening can progress without instant production breakage.
- Operational follow-up is required to review report-only violations before full enforce.
- 2026-03-04 update
- nonce 전파(`x-nonce`) 및 JSON-LD nonce 적용 자체는 검증했고 `/posts/[id]/guest`를 nonce 주입 가능 경로로 전환했다.
- 2026-04-04 correction
- production runtime audit 결과, Next.js inline bootstrap/hydration 제약 때문에 enforced CSP는 여전히 hydration-safe fallback(`script-src 'self' 'unsafe-inline'`)을 유지하고 strict nonce 정책은 report-only에만 적용된다.
- 따라서 `CSP_ENFORCE_STRICT`는 현재 배포 필수값이나 PASS gate로 취급하지 않는다.
- 2026-04-04 follow-up
- `connection()` 기반 nonce propagation과 framework inline style 2종(`color:transparent`, `position: absolute;`)의 exact SHA-256 allowlist를 적용해 `CSP_ENFORCE_STRICT=1` 경로를 다시 검증했다.
- production build + headless browser smoke에서 strict nonce CSP(`script-src 'self' 'nonce-...' 'strict-dynamic'`)가 실제 enforce되고 hydration/console 오류 없이 `/feed`가 렌더되는 것을 확인했다.
- `CSP_ENFORCE_STRICT`는 여전히 env validation의 hard requirement는 아니지만, production hardening 관점에서는 권장 설정으로 유지한다.

## ADR-SEC-006: Multi-window login throttling strategy
- Date: 2026-02-26
- Status: accepted
- Context
- Single-dimension IP rate limiting is insufficient against credential stuffing and distributed abuse.
- Decision
- Apply credentials login throttling across three dimensions (`ip`, `account+ip`, `account`) with different windows.
- Consequences
- Attackers need to evade multiple independent thresholds.
- Legitimate users may hit temporary friction after repeated failures; thresholds should be monitored and tuned.

## ADR-SEC-007: Strong-password policy with login compatibility
- Date: 2026-02-26
- Status: accepted
- Context
- Password policy needed strengthening, but existing users with legacy weak passwords still need to log in.
- Decision
- Apply strong policy to password set/update/reset boundaries, while keeping login validation compatibility-focused.
- Consequences
- New and rotated passwords become stronger immediately.
- Legacy weak passwords remain temporarily possible until users rotate credentials.

## ADR-SEC-008: Normalize duplicate-register signals
- Date: 2026-02-26
- Status: accepted
- Context
- Returning distinct `EMAIL_TAKEN` or `NICKNAME_TAKEN` responses can help enumeration attempts.
- Decision
- Map duplicate register conflicts to a single public response contract (`400 REGISTER_REJECTED`).
- Consequences
- Attackers receive less precise existence signals.
- UX gets less specific duplicate guidance at the API layer.

## ADR-SEC-009: Pepper-first guest hash with legacy compatibility
- Date: 2026-02-26
- Status: accepted
- Context
- Guest identity hashes were plain SHA-256, increasing re-identification risk if hash material leaks.
- Decision
- Use `GUEST_HASH_PEPPER` HMAC-SHA256 as primary hash and keep legacy SHA-256 as fallback candidate during lookup/matching.
- Consequences
- New writes gain stronger resistance to offline correlation.
- Backward compatibility is preserved for existing guest records without one-shot migration.
