# 돌봄 운영 런북

케어 요청 기능을 로컬에서 반복 점검할 때 사용하는 운영 절차입니다. 이 런북은 결제/보험/정산 없이 커뮤니티 매칭과 운영 이슈 신호만 확인합니다.

## 로컬 데이터 준비

기본 복구 루틴은 케어 데모 데이터를 함께 만듭니다.

```bash
corepack pnpm -C app db:restore:local
```

케어 데모 데이터만 다시 만들 때:

```bash
corepack pnpm -C app db:seed:care-demo
```

생성 계정:

- 요청자: `care.requester@townpet.dev`
- 지원자: `care.caregiver@townpet.dev`
- 운영자: `care.admin@townpet.dev`
- 기본 비밀번호: `SEED_DEFAULT_PASSWORD` 또는 `townpet123`

생성 데이터:

- `[CARE DEMO] 산책 요청 OPEN`: 피드/상세/지원 버튼 확인용
- `[CARE DEMO] 방문 돌봄 MATCHED`: 수락 지원자와 매칭 상태 확인용
- `[CARE DEMO] 완료 이슈 신호`: 완료 피드백 이슈 큐 확인용

## 수동 점검 순서

1. 게스트 피드에서 케어 요청 노출 확인
   - `/feed/guest?type=CARE_REQUEST&page=1`
   - 기대: `CARE_REQUEST` 글만 보이고, 지원/상태 전환 CTA는 로그인 요구로 막힌다.
2. 회원 피드와 상세 확인
   - 요청자 또는 지원자로 로그인 후 `/feed?type=CARE_REQUEST&page=1`
   - `[CARE DEMO] 산책 요청 OPEN` 상세에서 돌봄 메타와 지원 CTA를 확인한다.
3. 지원/매칭 상태 확인
   - `[CARE DEMO] 방문 돌봄 MATCHED` 상세에서 수락 지원자와 상태 뱃지를 확인한다.
4. 완료 피드백 이슈 큐 확인
   - 운영자로 로그인 후 `/admin/care-feedbacks`
   - 기대: `SAFETY` issue type의 `[CARE DEMO] 완료 이슈 신호`가 보인다.
   - 처리 상태를 `검토중` 또는 `해결`로 바꾸고 운영자 메모를 저장한다.
5. 운영 overview 확인
   - `/admin/ops`
   - 기대: 돌봄 이슈 신호 카드가 1건 이상을 표시한다.

## 자동화 검증

로컬 서버가 켜진 상태에서 핵심 브라우저 흐름을 확인합니다.

```

문서/타입/단위 회귀는 아래 순서로 확인합니다.

corepack pnpm -C app typecheck
corepack pnpm -C app lint
corepack pnpm -C app docs:refresh:check
```

- 버그: 재현 URL, 계정, 클릭 단계, 기대/실제 결과, 원인 후보를 `docs/PROGRESS.md`에 남긴다.
- 보류: 결제/보험/정산, 자동 제재, 증빙 업로드, production smoke는 이번 런북 범위 밖이다.

## 출시 판단 요약

- 로컬 기능 판정: 조건부 Go. M1-M12로 작성, 지원, 매칭, 수행 상태, 완료 피드백, 관리자 큐, 모바일 운영, Ops threshold까지 확인 가능하다.
- 운영 배포 판정: No-Go. production smoke, 운영 secret 기반 strict health, 실 운영 계정/알림 확인은 아직 별도 검증이 필요하다.
- 포함 범위: 결제 없는 커뮤니티 매칭, 비공개 완료 피드백, 운영 이슈 신호 큐, 수동 처리 메모, Ops threshold.
- 보류 범위: 결제/보험/정산, 자동 제재/자동 숨김, 증빙 업로드, 공개 평판 점수, production smoke.
- 다음 확인: `db:restore:local`, care e2e, 모바일 큐 e2e, `quality:check`, 운영 secret을 주입한 원격 health/smoke 순서로 진행한다.

## Production Smoke Preflight

운영 배포 smoke는 아래 조건이 준비되기 전까지 실행하지 않는다.

필수 secret/env:
- Vercel runtime: `DATABASE_URL`, `AUTH_SECRET`, `APP_BASE_URL`, `NEXTAUTH_URL`, `GUEST_HASH_PEPPER`, `HEALTH_INTERNAL_TOKEN`, `UPSTASH_REDIS_REST_URL`, `UPSTASH_REDIS_REST_TOKEN`, `RESEND_API_KEY`, `BLOB_READ_WRITE_TOKEN`
- GitHub Actions: `HEALTH_INTERNAL_TOKEN`
- 선택 검증: `SENTRY_DSN`, `SENTRY_AUTH_TOKEN`, `SENTRY_ORG_SLUG`, `SENTRY_PROJECT_SLUG`

실행 순서:

```bash
corepack pnpm -C app quality:check
OPS_BASE_URL=https://townpet.vercel.app corepack pnpm -C app ops:check:health
OPS_BASE_URL=https://townpet.vercel.app OPS_HEALTH_INTERNAL_TOKEN=<HEALTH_INTERNAL_TOKEN> OPS_HEALTH_REQUIRE_PG_TRGM=1 corepack pnpm -C app ops:check:health
OPS_BASE_URL=https://townpet.vercel.app OPS_EVIDENCE_SECURITY_STRICT=1 corepack pnpm -C app ops:evidence
```

GitHub Actions 수동 실행:
- `ops-smoke-checks`
  - `target_base_url=https://townpet.vercel.app`
  - `verify_pg_trgm=true`
  - `verify_sentry=true`는 Sentry secret이 모두 준비된 경우에만 사용한다.

브라우저 수동 smoke:
1. `/feed/guest?type=CARE_REQUEST&page=1`
   - `CARE_REQUEST` 글 노출과 게스트 지원 차단을 확인한다.
2. `/feed?type=CARE_REQUEST&page=1`
   - 운영자가 준비한 테스트 회원으로 로그인 후 돌봄 메타와 지원 CTA를 확인한다.
3. `/admin/care-feedbacks`
   - 운영자 계정으로 이슈 큐, 처리 상태 필터, 메모 저장을 확인한다.
4. `/admin/ops`
   - 돌봄 이슈 count, threshold badge/copy가 표시되는지 확인한다.

No-Go 중단 기준:
- public 또는 internal `/api/health`가 `ok`가 아니다.
- `OPS_HEALTH_REQUIRE_PG_TRGM=1` 확인이 실패한다.
- 운영자 계정으로 `/admin/care-feedbacks` 또는 `/admin/ops` 접근이 안 된다.
- 케어 요청이 공개 피드/상세에서 깨지거나, 관리자 큐 저장이 실패한다.
- production smoke 중 실 사용자 데이터에 테스트 피드백/메모를 남겨야 하는 상황이 생긴다.

## 운영 계정/데이터 Smoke 기준

운영 smoke는 실 사용자 데이터 변경을 피하는 것이 기본값이다. 테스트 데이터 생성이 꼭 필요하면 아래 조건을 모두 만족해야 한다.

계정 조건:
- 관리자: 실제 운영 관리자 1명만 사용하고, `/admin/care-feedbacks`, `/admin/ops` 접근 권한을 smoke 전에 확인한다.
- 요청자 테스트 회원: 운영자가 소유한 전용 테스트 계정만 사용한다. 이메일/닉네임은 smoke용임을 식별할 수 있어야 한다.
- 지원자 테스트 회원: 요청자와 다른 전용 테스트 계정만 사용한다. 실 사용자 계정으로 지원/피드백을 만들지 않는다.
- 게스트: 조회와 로그인 요구 확인만 수행한다. 게스트로 테스트 글, 댓글, 신고, 피드백을 만들지 않는다.

데이터 생성 원칙:
- 기존 실사용자 케어 요청에는 지원, 완료 피드백, 운영자 메모를 남기지 않는다.
- production에서 새 테스트 글이 필요하면 제목은 `[SMOKE][CARE] yyyy-mm-dd ...` 형식으로 만든다.
- 테스트 글 본문에는 실제 연락처, 주소, 결제 정보, 반려동물 민감 정보를 넣지 않는다.
- 완료 피드백 이슈 신호가 필요하면 테스트 요청자/지원자 조합으로만 만들고, issue type은 `OTHER`를 우선한다.
- `SAFETY`, `PAYMENT_OR_FRAUD`는 실제 고위험 경고 검증이 필요한 경우에만 쓰고 smoke 직후 정리한다.

Smoke 후 정리:
- 테스트 케어 요청은 작성자 또는 운영자가 `CANCELLED` 상태로 정리한다.
- 테스트 완료 피드백에 남긴 운영자 메모는 `[SMOKE 정리 완료]`로 닫거나 `DISMISSED`로 전환한다.
- `/admin/ops`에서 테스트 이슈 count가 운영 판단을 오염시키지 않는지 확인한다.
- `ModerationActionLog`에 테스트 상태 변경과 피드백 검토 로그가 남는지 확인한다.
- 정리할 수 없는 데이터가 생기면 즉시 No-Go로 기록하고 추가 smoke를 중단한다.
