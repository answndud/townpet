## 최소 운영 기준

- 일상 명령은 `pnpm -C app quality:check`, `pnpm -C app ops:check:health`, `pnpm -C app db:restore:local` 세 개만 기본으로 둡니다.
- 1인 운영 production smoke는 기본적으로 `OPS_BASE_URL=https://townpet.vercel.app pnpm -C app ops:evidence:solo`만 실행합니다.
- 배포 전에는 `배포전_on-demand_체크.md`에 따라 `quality:check`를 먼저 통과시키고, 변경 범위에 맞는 e2e만 추가 실행한 뒤, 배포 후 `ops:check:health`로 생존 확인합니다. 상세/댓글/피드 item/권한형 상세 UI를 바꾼 배포는 health 다음에 `ops:check:detail-visual`까지 실행합니다.
- 관리자 브라우저 확인은 인증/관리자/신고/정정/정책/DB migration/env 변경이 있을 때만 실행합니다. 신고/정정 큐 변경은 가능하면 `ADMIN_QUEUE_SMOKE_EMAIL`, `ADMIN_QUEUE_SMOKE_PASSWORD`를 설정한 뒤 `pnpm -C app ops:check:admin-queue-smoke`로 확인합니다. GitHub Actions에서는 `ops-smoke-checks`를 `verify_admin_queue=true`로 수동 실행하면 같은 secret으로 authenticated admin queue smoke를 실행합니다. 이 workflow는 `app/.playwright-browsers` cache를 사용하므로 첫 실행은 브라우저 설치 때문에 느릴 수 있고, 이후 실행은 더 짧아야 합니다. production credential이 없으면 `ADMIN_QUEUE_SMOKE_LOCAL_FIXTURES=1 OPS_BASE_URL=http://localhost:3000` 로컬 fixture 모드로 UI/큐 렌더링만 확인하고, production 인증 smoke는 blocker로 기록합니다.
- 먼저 보는 자동화는 `.github/workflows/quality-gate.yml`, `.github/workflows/ops-smoke-checks.yml` 두 개입니다.
- 브라우저 smoke는 `.github/workflows/browser-smoke.yml` 또는 로컬 `pnpm -C app test:e2e:smoke`로만 on-demand 실행합니다. 현재 smoke는 로그인 진입/온보딩과 게시글 에디터 툴바 regression까지 함께 봅니다.
- 나머지 cleanup/backfill/manual-report/latency/post-integrity workflow는 자동 schedule 없이 필요할 때만 `workflow_dispatch`로 여는 유지보수 도구입니다.

### 1인 운영 solo evidence runner

기본 로컬 실행:

```bash
pnpm -C app ops:evidence:solo
```

원격 배포 기준 실행:

```bash
OPS_BASE_URL=https://townpet.vercel.app pnpm -C app ops:evidence:solo
```

판정 기준:
- 전체 status가 `PASS`면 public `/api/health`가 정상인 상태입니다.
- 실패하면 runner는 exit code 1로 종료하고 `docs/reports/ops-evidence-*.md`에 실패 증적을 남깁니다.
- 생성되는 `docs/reports/ops-evidence-*.md`는 로컬 evidence라 커밋하지 않습니다.

### On-demand 확장 evidence

health 외에 security env, prewarm, latency snapshot까지 확인할 때만 실행합니다.

```bash
OPS_BASE_URL=https://townpet.vercel.app pnpm -C app ops:evidence
```

- production strict env까지 확인할 때만 `OPS_EVIDENCE_SECURITY_STRICT=1`을 추가합니다.

### On-demand maintenance workflows

아래 workflow는 production DB 또는 외부 배포를 대상으로 하므로 1인 운영 기본 루틴에서 제외합니다. 데이터 증가, 정합성 drift, 장애 조사처럼 명확한 이유가 있을 때만 GitHub Actions에서 수동 실행합니다.

- `.github/workflows/auth-audit-cleanup.yml`
  - 인증 감사 로그 보존기간 정리가 필요할 때 실행
- `.github/workflows/notification-cleanup.yml`
  - 보관 알림 정리가 필요할 때 실행
- `pnpm -C app ops:notifications:retry -- --dry-run`
  - 알림 outbox의 `PENDING/FAILED` 누적이나 알림 생성 실패가 의심될 때 먼저 dry-run으로 대기/실패/재처리 대상 수를 확인
  - 실제 재처리는 `pnpm -C app ops:notifications:retry -- --apply`로 실행하며, 기본 limit은 `NOTIFICATION_OUTBOX_RETRY_LIMIT`로 조정
- `.github/workflows/search-term-cleanup.yml`
  - 검색 통계 테이블 크기나 오래된 검색어 정리가 필요할 때 실행
- `.github/workflows/post-integrity-maintenance.yml`
  - 삭제 글/알림/카운트 정합성 drift가 의심될 때 dry-run 먼저 실행
- `.github/workflows/ops-latency-snapshots.yml`
  - 성능 회귀 의심 또는 배포 전후 비교가 필요할 때 실행
- `.github/workflows/production-demo-content.yml`
  - 운영 DB에 데모 콘텐츠를 만들거나 지우는 legacy helper다. 일반 운영에서는 실행하지 않고, 소유한 demo 도메인과 정리 계획이 있을 때만 수동 실행한다.
- `pnpm -C app ops:operator-content:publish`
  - 첫 7개 전국 공통 운영자 정리 글을 게시한다. 기본은 dry-run이며, non-local DB에서는 `OPERATOR_CONTENT_PUBLISH_CONFIRM=PUBLISH_OPERATOR_CONTENT`, 실제 생성은 추가로 `OPERATOR_CONTENT_PUBLISH_APPLY=1`이 필요하다.
- `pnpm -C app ops:public-smoke-fixtures:publish`
  - public 상세 smoke에 필요한 `LOST_FOUND`, `MARKET_LISTING` 운영자 fixture를 게시한다. 기본은 dry-run이며, non-local DB에서는 `PUBLIC_SMOKE_FIXTURE_CONFIRM=PUBLISH_PUBLIC_SMOKE_FIXTURES`, 실제 생성은 추가로 `PUBLIC_SMOKE_FIXTURE_APPLY=1`이 필요하다.
- `pnpm -C app ops:check:operator-content-public`
  - 운영자 정리 글 게시 후 `/api/feed/guest`, 상세, `/feed/guest`, `/search/guest`, `/api/home/feed` public 노출을 확인한다. 글이 아직 없으면 `BLOCKED`로 종료한다.
- `pnpm -C app ops:check:public-detail-visual`
  - production public feed에서 type별 상세 target을 찾고 desktop/mobile screenshot smoke를 실행한다. 실제 public target이 없는 type은 report에 `BLOCKED`로 남긴다.
- `pnpm -C app ops:check:auth-local-detail-visual`
  - guest/public smoke로 열 수 없는 `HOSPITAL_REVIEW`, `CARE_REQUEST` 상세를 로그인/동네 컨텍스트로 검증한다. non-local DB에서는 `AUTH_LOCAL_DETAIL_SMOKE_CONFIRM=PUBLISH_AUTH_LOCAL_DETAIL_SMOKE_FIXTURES` 확인값이 필요하고, 실행 시 검증용 사용자/게시글을 upsert한 뒤 desktop/mobile screenshot report를 남긴다.
  - 같은 report 안에 guest 접근 negative check도 남긴다. `/posts/{id}/guest`에서는 로그인 안내 gate가 보이고 보호 대상 상세 제목이 노출되지 않아야 한다.
- `pnpm -C app ops:check:detail-visual`
  - 배포 후 상세 화면 시각 smoke를 한 번에 실행한다. 순서는 `health -> public detail visual -> auth/local detail visual`이다.
  - non-local DB에서 전체 실행하려면 production DB env와 `AUTH_LOCAL_DETAIL_SMOKE_CONFIRM=PUBLISH_AUTH_LOCAL_DETAIL_SMOKE_FIXTURES`가 필요하다.
  - auth/local DB upsert 없이 public 상세만 볼 때는 `DETAIL_VISUAL_SMOKE_SKIP_AUTH_LOCAL=1`을 명시한다.
  - 실행 결과는 `docs/reports/detail-visual-smoke-*.md`에 남고, 개별 screenshot은 각 smoke script의 `docs/reports/*visual-smoke-*` 폴더에 남는다.
- `pnpm -C app db:audit:demo-content`
  - production demo/E2E 의심 데이터를 찾는 read-only audit이다. non-local DB에서는 `DEMO_CONTENT_AUDIT_CONFIRM=DEMO_CONTENT_AUDIT` 확인값이 필요하다.
- `pnpm -C app db:audit:legacy-upload-paths`
  - production 본문/댓글/PostImage에 남은 `/media/media/uploads/*` 중복 업로드 경로를 찾는 read-only audit이다. non-local DB에서는 `LEGACY_UPLOAD_PATH_AUDIT_CONFIRM=LEGACY_UPLOAD_PATH_AUDIT` 확인값이 필요하다.

로컬 cleanup/backfill/repair 스크립트는 기본값이 dry-run입니다. 데이터 변경이 필요한 경우에만 `--apply`를 명시하고, GitHub Actions maintenance workflow는 workflow input 또는 내부 command에서 apply 모드를 명시합니다.

## 운영 화면 빠른 지도

- `/admin/reports`
  - 신고 큐, auto-hide, 제재 적용 상태 확인
- `/admin/care-feedbacks`
  - 돌봄 완료 피드백 이슈 신호 확인, 로컬 재현 절차는 `돌봄_운영_런북.md`를 따른다
- `/admin/auth-audits`
  - 로그인/비밀번호/회원가입 감사 로그 확인
- `/admin/policies`
  - 운영 정책과 개인화 튜닝 정책 조정
- `/admin/personalization`
  - personalized feed CTR, 광고 반응, audience key 지표 확인
  - 지표별 판정과 조치 기준은 `개인화_운영_판단_기준.md`를 따른다
- `/admin/breeds`
  - 품종 사전 override/custom row 관리
- `/my-posts`
  - 사용자 작성글 목록 확인, 지원 응대 시 작성 이력 검토
- `/bookmarks`
  - 사용자 북마크 글 확인, 지원 응대 시 북마크 상태 검토 (`/saved`는 호환 리다이렉트)
- `/users/{userId}`
  - 로그인 사용자만 접근 가능한 공개 프로필, 공개 범위 설정에 따라 게시글/댓글/반려동물 노출 상태 확인
