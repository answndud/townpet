# free-ops-weekly-checklist.md

# 무료 운영 체크리스트 (주간 10분 루틴)

목표
- 외부 SaaS를 최소화한 상태에서, 배포 실패/장애를 빠르게 감지하고 기본 운영 품질을 유지한다.

원칙
- 기본 모니터링: `GitHub Actions + /api/health + Vercel 로그`
- CSP 위반은 `GET /api/security/csp-report` 내부 조회로 주기 확인
- Sentry는 선택사항(기본 OFF)
- 매주 1회, 10분 안에 끝내기

---

## 0-2분: 배포 상태 확인

1. Vercel 프로젝트의 최근 배포 3건 확인
2. 실패 배포가 있으면 빌드 로그에서 첫 에러 원인만 기록

완료 기준
- 최근 배포가 `Ready`이거나, 실패 원인이 메모되어 있음

---

## 2-4분: Health 스모크 실행

로컬 evidence runner 권장:
- `OPS_BASE_URL=https://townpet.vercel.app pnpm -C app ops:evidence`
- 결과 파일: `docs/reports/ops-evidence-*.md`

GitHub Actions에서 아래 워크플로우 실행
- Workflow: `ops-smoke-checks`
- 입력
  - `target_base_url`: `https://townpet.vercel.app`
  - `verify_sentry`: `false` (기본값)

완료 기준
- `ops:evidence` 전체 status PASS 또는 `Check deployment health endpoint` 단계 PASS

---

## 4-6분: 오류 로그 스팟체크

1. Vercel Functions 로그 최근 24시간 확인
2. `500`, `Prisma`, `timeout`, `Unhandled` 키워드로 빠르게 훑기
3. 같은 오류가 3회 이상 반복되면 이슈 1건 생성

완료 기준
- 반복 오류 없음 또는 이슈 생성 완료

---

## 6-7분: CSP 위반 통계 점검

사전 준비
- 로컬 셸에 토큰 설정
  - `export HEALTH_INTERNAL_TOKEN='<운영 토큰>'`

조회 명령
- `curl -sS -H "Authorization: Bearer $HEALTH_INTERNAL_TOKEN" https://townpet.vercel.app/api/security/csp-report | jq`

확인 포인트
- `data.totalBuckets` 증가 추이 확인
- `data.top` 상위 3건의 `effectiveDirective`, `blockedUri`, `documentUri` 확인
- 같은 위반이 연속 증가하면 티켓 생성 (`SEC-002` 연계)

완료 기준
- 상위 위반 3건이 기록되었거나, 위반 없음이 기록됨

---

## 7-9분: 사용자 핵심 동선 1회 수동 점검

1. `/feed` 접속
2. 게시글 상세 진입 후 `목록으로` 복귀
3. 글쓰기 화면 진입 확인

완료 기준
- 주요 동선 3단계 이상 없음

---

## 9-10분: 기록

`PROGRESS.md` 하단에 아래 3가지만 append
- 실행 일시
- `ops:evidence` 로컬 결과 파일 또는 `ops-smoke-checks` run URL + PASS/FAIL
- CSP 상위 위반 3건 요약(없으면 `없음`)

주간 기록 템플릿
```md
### YYYY-MM-DD weekly ops check
- ops-evidence: docs/reports/ops-evidence-<timestamp>.md (PASS/FAIL)
- ops-smoke-checks: <run-url or 미실행> (PASS/FAIL/미실행)
- CSP top3:
  - <effectiveDirective> | <blockedUri> | count=<n>
  - <effectiveDirective> | <blockedUri> | count=<n>
  - <effectiveDirective> | <blockedUri> | count=<n>
- issue: <없음 또는 링크>
```

---

## 월간(선택, 10분)

- `verify_sentry=true`로 1회 실행(원할 때만)
- 실패 시 Sentry 시크릿 미등록 상태로 유지하고 continue

비고
- 현재 기본 운영은 Sentry 없이도 가능하도록 설계됨
- Sentry는 "필수"가 아니라 "문제 추적이 어려워질 때" 도입
