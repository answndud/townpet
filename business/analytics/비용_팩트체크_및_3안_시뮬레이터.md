# Next.js + Vercel 비용 팩트체크 및 3안 시뮬레이터

기준일: 2026-02-24  
대상: 커뮤니티/SNS 초기~성장 구간의 인프라 비용 판단

## 1) 한 줄 결론
- "유저가 늘면 비용이 무조건 기하급수"는 과장이다.
- 실제로는 대체로 `선형`에 가깝고, 캐시/이미지/봇 방어가 약할 때 특정 축이 급증한다.

## 2) 팩트체크 (무엇이 진짜 비용을 키우는가)
- 유저 수 자체보다 `요청 수`, `전송량`, `이미지/미디어`, `함수 실행 시간`, `DB 쿼리 효율`이 비용을 결정한다.
- 커뮤니티에서 가장 자주 튀는 항목은 이미지 경로(`/_next/image`)와 캐시 미스 SSR/API 경로다.
- 폭증 체감은 보통 아래 3개 동시 발생에서 온다.
  - 이미지 리사이징/캐시 미스 증가
  - CDN 전송량(특히 모바일 이미지) 증가
  - 봇/스크래퍼 트래픽 방치

## 3) Vercel 과금 축 (공식 문서 기준)
아래는 공개 문서에 있는 대표 단가/포함량(지역/시점에 따라 변동 가능).

- Pro 기본: `$20 / user / month` + 사용량 과금 (`$20` 포함 크레딧)
- Edge Requests 포함량: Pro `10M / month`
- Fast Data Transfer 포함량: Pro `1TB / month`
- Functions Invocations: Pro 기준 `1M` 포함, 초과 `~$0.60 / 1M`
- Functions Active CPU: 초과 `~$0.128 / hour`
- Functions Provisioned Memory: 초과 `~$0.0106 / GB-hour`
- Image Transformations: 초과 `~$0.05 / 1K`
- Image Cache Reads: 초과 `~$0.40 / 1M`

참고:
- https://vercel.com/pricing
- https://vercel.com/docs/functions/usage-and-pricing
- https://vercel.com/docs/manage-cdn-usage

## 4) 월 비용 계산식 (실무형)
총비용은 아래처럼 분해해 보는 것이 가장 정확하다.

```text
월비용 = 팀고정비(플랜/좌석)
      + 함수비용(Invocations + ActiveCPU + Memory)
      + 네트워크비용(Edge Requests + Fast Data Transfer)
      + 이미지비용(Transform + Cache Read/Write)
      + DB비용(컴퓨트 + 스토리지 + 백업)
      + 기타(Sentry/로그/메일/SMS)
```

## 5) 3안 시뮬레이션 (가정 기반)
아래는 "초기 커뮤니티"를 가정한 러프 모델이다. 정확 산출은 실제 대시보드 수치로 보정해야 한다.

가정값:
- 월 페이지뷰(PV): 시나리오별 상이
- 페이지당 함수 호출: `1.5`
- 함수 평균 Active CPU: `40ms`
- 페이지당 이미지 요청: `4`
- 이미지 변환 캐시 미스율: `5%`
- 페이지 평균 전송량(HTML+JS+이미지 합): `0.7MB`

| 시나리오 | 월 PV | 함수 호출(추정) | 이미지 변환(추정) | 전송량(추정) | 판단 |
|---|---:|---:|---:|---:|---|
| 낙관(초기) | 300,000 | 450,000 | 60,000 | ~210GB | Pro 포함량 안쪽 가능성 높음 |
| 기준(성장) | 1,500,000 | 2,250,000 | 300,000 | ~1.05TB | 전송량부터 초과 가능성 시작 |
| 비관(급성장) | 7,000,000 | 10,500,000 | 1,400,000 | ~4.9TB | 네트워크/이미지/봇 대응 필수 |

해석:
- 함수비용은 생각보다 완만한 편이고, 전송량/이미지가 먼저 지갑을 때리는 경우가 많다.
- 즉, "백엔드 함수 최적화"보다 "캐시 + 이미지 + 봇 차단"이 초기 비용 방어에 더 효과적이다.

## 6) 비용 급증 방지 체크리스트 (우선순위)
- `/_next/image` 상위 경로 모니터링: 과다 호출 페이지 우선 정리
- 이미지 정책: 원본 업로드 제한, 썸네일 규격 고정, 고해상도 지연 로딩
- 캐시 정책: 정적/ISR 우선, API 응답 캐시 헤더 점검
- DB 쿼리: N+1 제거, 인덱스 보강, 리스트 페이지 필드 최소화
- 봇 대응: 레이트리밋, WAF 룰, 신규 유저/익명 엔드포인트 제한
- 예산 보호: Vercel Spend Management 알림/하드리밋/자동 pause 설정

## 7) 대안 아키텍처 (비용 관점)
### A. Vercel 유지 (권장 출발)
- 장점: 운영 복잡도 최소, 배포 생산성 높음
- 단점: 트래픽 폭증 시 네트워크/이미지 과금 체감
- 추천 시점: PMF 탐색~초기 성장

### B. 하이브리드 분리
- 구성: 프론트/캐시는 Vercel, 무거운 API/워커/크롤러는 별도 컴퓨트
- 장점: 비싼 구간만 분리해 비용 탄력 확보
- 단점: 운영 복잡도 증가(모니터링/배포/장애면)
- 추천 시점: 월 비용에서 특정 워크로드가 30%+ 차지할 때

### C. 전체 이전(셀프 호스팅 포함)
- 장점: 단가 최적화 여지
- 단점: 장애/보안/스케일링 책임 증가
- 추천 시점: 트래픽 패턴이 안정적이고 운영 인력이 확보된 이후

## 8) 지금 바로 적용할 실행안
- 1주차: 사용량 대시보드에서 상위 10개 경로(요청/전송/실행) 확정
- 2주차: 이미지 규격/캐시 정책 일괄 적용
