# TownPet (동네 반려 커뮤니티/마켓/구인구직) — 이력서 기재용 프로젝트 소개

## 1) 프로젝트 한 줄 요약
- **TownPet**: 동네 기반으로 **동물병원 리뷰(비용/대기/서비스)**, **산책 코스**, **사료/용품 정보**를 공유하고, **중고/대여 마켓**과 **펫케어 구인구직**까지 제공하는 로컬 커뮤니티 플랫폼

## 2) 문제 정의(왜 만들었나)
- 동네 반려인들이 실제로 필요한 정보(병원비, 대기시간, 산책 코스 위험 요소, 사료 알러지 경험 등)는 흩어져 있고 최신성이 떨어짐
- 커뮤니티/마켓/구인을 한 곳에 모을 경우 **스팸/허위정보/분쟁**이 빠르게 증가하므로, “기능 구현”보다 **운영 정책 + 신뢰 시스템 + 성능**이 중요

## 3) 핵심 기능(구현 범위)
### 로컬(동네) 기능
- 동네 피드(대표 동네 + 인접 동네 범위)
- **동물병원 리뷰/비용 공유**
  - 템플릿 기반 작성(방문일/진료항목/총액/대기시간/영수증 여부)
  - 병원 프로필에서 항목별 비용/대기시간 집계(익명 집계)
- **산책 코스 공유**
  - 지도 기반 위치/코스 정보 + 안전 태그(차량/자전거/결빙 등)

### 글로벌(지역 무관) 기능
- 견종/사료 등 토픽 기반 게시판 및 Q&A(선택 구현)

### 마켓/구인구직
- 중고/대여 마켓(대여: 보증금/기간/인수·반납 체크리스트)
- 펫케어 구인구직(공고 템플릿, 지원/상태 관리)

### 운영/안전(핵심)
- 신고/차단/자동 블라인드(룰 기반)
- 스팸/어뷰징 방지(레이트리밋, 신규 계정 제한, 링크/연락처 노출 제한)

## 4) 역할 및 기여(백엔드 중심, 본인 기여 예시)
- **Spring Boot 기반 API 설계 및 구현**(게시물/댓글/검색/마켓/구인/신고 등)
- **도메인 중심 모듈러 모놀리스 구조**로 설계하여 기능 확장 및 분리 가능성 확보
- **피드/검색 성능 최적화**
  - 커서 기반 페이지네이션 적용, 복합 인덱스 설계
  - QueryDSL로 동적 필터/정렬 조립
- **운영 정책 엔진 구현**
  - 신고 누적에 따른 자동 블라인드/재노출 기준
  - 관리자 조치 이력(Audit Log) 기록
- **Redis 기반 레이트리밋/스팸 방지**
  - 글/댓글/검색/로그인 엔드포인트에 rate limit 적용
- **비동기 이벤트 처리(선택 구현)**
  - 알림 발송/검색 인덱싱/이미지 후처리를 이벤트 기반으로 분리(메시지 큐)
- **통합 테스트/배포 자동화**
  - Testcontainers(Postgres/Redis)로 통합 테스트 구성
  - GitHub Actions + Docker 기반 CI/CD 파이프라인 구축

## 5) 사용 기술 스택
### Backend
- Java 17/21, Spring Boot 3.x, Spring Security(OAuth2 + JWT)
- Spring Data JPA(Hibernate), **QueryDSL**
- Flyway(DDL 마이그레이션)
- Redis(캐시/레이트리밋/어뷰징 방지)
- 메시지 큐(선택): RabbitMQ 또는 Kafka
- Observability(선택): Micrometer(+ Prometheus/Grafana), Sentry

### Data / Storage
- PostgreSQL
- S3 호환 스토리지(AWS S3 / Cloudflare R2) + presigned upload

### Front(연동)
- Spring Boot MVC + Thymeleaf(SSR)
- HTMX(부분 갱신) + Tailwind
- (선택) Alpine.js: 모달/드롭다운 등 최소 UI
- 지도: 카카오맵/네이버맵 SDK(필요 화면에서만 최소 JS)

### Infra / DevOps
- Docker
- GitHub Actions(CI/CD)
- 배포: AWS ECS/Fargate(또는 Fly.io/Render 등)

## 6) 성능/품질 포인트(면접에서 말하기 좋은 근거)
- **Cursor Pagination**으로 대규모 피드에서 offset paging 병목 회피
- 동네/타입/생성일 기반 **복합 인덱스**로 피드/검색 응답 개선
- QueryDSL로 다양한 필터 조건을 안전하게 조립(동네, 태그, 가격대, 시간 등)
- Redis 토큰버킷/슬라이딩 윈도우 방식으로 rate limit 적용(어뷰징 방지)
- 신고/블라인드 정책을 코드로 모델링(운영 자동화)
- 통합 테스트 환경을 컨테이너로 고정해 재현성 확보(Testcontainers)

## 7) 트러블슈팅(면접 답변용 예시 3개)
1) **피드 성능 이슈**
   - 문제: 동네/타입/정렬 조건이 많아지며 쿼리 비용 증가
   - 대응: 커서 기반 페이지네이션 + 복합 인덱스 + 조건 분리(QueryDSL)로 개선
2) **스팸/업자 계정 유입**
   - 문제: 구인/마켓에 링크/연락처를 통한 스팸 급증
   - 대응: 레이트리밋 + 신규 계정 제약 + 링크 노출 제한 + 신고 자동 블라인드
3) **운영(신고) 처리 비용 증가**
   - 문제: 신고 처리의 수동 대응 비용 증가
   - 대응: 신고 기준 자동 블라인드 + 관리자 조치 Audit Log + 재노출 규칙 정의
