# GuestAuthor 모델 분리 마이그레이션 계획

기준일: 2026-02-24

## 배경
- 현재 비회원 글/댓글 작성 시 `User` 레코드를 생성해 작성자를 연결한다.
- 이 구조는 `User` 테이블에 게스트 계정이 지속 누적되어 운영/분석/성능 측면에서 분리가 어렵다.

## 목표
- 비회원 작성자 식별을 `User`에서 분리해 `GuestAuthor` 기반으로 관리한다.
- 게시글/댓글의 작성자 참조를 `authorId(User)`와 `guestAuthorId(GuestAuthor)` 이중 구조로 전환한다.
- 기존 데이터 호환을 위한 점진 이관 경로를 제공한다.

## 제안 스키마 (초안)

```prisma
model GuestAuthor {
  id              String   @id @default(cuid())
  displayName     String
  passwordHash    String
  ipHash          String
  fingerprintHash String?
  ipDisplay       String?
  ipLabel         String?
  createdAt       DateTime @default(now())
  updatedAt       DateTime @updatedAt

  posts           Post[]
  comments        Comment[]

  @@index([ipHash, createdAt(sort: Desc)])
  @@index([fingerprintHash, createdAt(sort: Desc)])
}

model Post {
  // 기존
  authorId      String
  // 신규
  guestAuthorId String?

  author       User        @relation(fields: [authorId], references: [id])
  guestAuthor  GuestAuthor? @relation(fields: [guestAuthorId], references: [id], onDelete: SetNull)

  @@index([guestAuthorId, createdAt(sort: Desc)])
}

model Comment {
  // 기존
  authorId      String
  // 신규
  guestAuthorId String?

  author       User         @relation(fields: [authorId], references: [id])
  guestAuthor  GuestAuthor? @relation(fields: [guestAuthorId], references: [id], onDelete: SetNull)

  @@index([guestAuthorId, createdAt(sort: Desc)])
}
```

## 서비스 레이어 변경 방향
- `post.service.createPost`
  - 비회원 작성 시 `prisma.user.create` 제거.
  - `GuestAuthor` 생성 후 `guestAuthorId`로 연결.
  - `authorId`는 시스템 게스트 전용 사용자(`guest.system@townpet.local`)로 고정하는 이행안 검토.
- `comments/[id]/route` + `comment.service.createComment`
  - 댓글 비회원 작성도 동일하게 `GuestAuthor`로 연결.
- 조회 쿼리(`post.queries`, `comment.queries`)
  - 렌더링용 작성자 표시명은 `guestAuthor.displayName` 우선, 기존 `guestDisplayName` fallback 유지.

## 마이그레이션 단계
1. **Schema Expand**
   - `GuestAuthor` 추가, `Post/Comment.guestAuthorId` 추가.
   - 기존 guest 메타 컬럼(`guestDisplayName`, `guestPasswordHash`, `guestIpHash`...) 유지.
2. **Dual Write**
   - 신규 비회원 작성은 `GuestAuthor`와 기존 컬럼을 동시에 기록.
3. **Backfill**
   - 기존 guest post/comment에서 `GuestAuthor`를 생성/연결하는 스크립트 실행.
4. **Read Switch**
   - 조회 경로를 `guestAuthor` 우선으로 전환.
5. **Cleanup**
   - 안정화 후 중복 guest 메타 컬럼 제거(후속 사이클).

## 리스크 / 체크리스트
- 작성 권한/비밀번호 검증 로직이 `GuestAuthor.passwordHash` 기준으로 정상 동작하는지 회귀 테스트 필요.
- 차단/제재 키(`ipHash`, `fingerprintHash`)가 기존과 동일하게 집계되는지 확인.
- 레거시 데이터 백필 시 중복 `GuestAuthor` 생성 정책(완전 병합 vs 게시물 단위 1:1) 확정 필요.

## 테스트 계획
- 단위: `post.service`, `comment.service` 비회원 작성/수정/삭제 비밀번호 검증.
- 통합: `/api/posts`, `/api/posts/[id]/comments` 비회원 작성 계약 테스트.
- 데이터: 백필 스크립트 dry-run/적용 모드 분리, 처리 건수 로그 출력.

## 백필 실행 가이드
- dry-run: `cd app && GUEST_AUTHOR_BACKFILL_DRY_RUN=1 pnpm db:backfill:guest-authors`
- apply: `cd app && pnpm db:backfill:guest-authors`
- batch size 조정: `GUEST_AUTHOR_BACKFILL_BATCH_SIZE=500` (기본 200, 최대 1000)
