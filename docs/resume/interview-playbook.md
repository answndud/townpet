# 인터뷰 플레이북

## 1. 면접의 목표

이 면접에서 이겨야 할 프레임:

> 저는 AI로 산출물을 많이 만든 사람이 아니라, AI와 함께 문제를 쪼개고 결과를 검증하며 운영 가능한 형태로 닫는 사람입니다.

피해야 할 프레임:
- "반려동물 커뮤니티를 만들었습니다."
- "AI가 많이 도와줬습니다."
- "테스트 많이 돌렸습니다."

반드시 보여줄 프레임:
- 문제를 직접 정의했다.
- 기능을 사용자 여정과 백엔드 정책으로 분해했다.
- AI의 한계를 알고 검증 루프를 만들었다.
- 운영 중 생긴 문제를 재발 방지까지 닫았다.
- 없는 경험은 과장하지 않고 tradeoff를 말할 수 있다.

## 2. 30초 자기소개

저는 AI agent와 함께 실제 제품을 끝까지 만드는 방식으로 성장해 온 지원자입니다. 대표 프로젝트인 TownPet은 지역 기반 반려생활 정보 커뮤니티이지만, 단순 게시판이 아니라 구조화 게시판, 비회원 안전장치, 신고/제재, 검색 품질, 알림, 배포 preflight, production evidence까지 포함한 서비스형 백엔드 포트폴리오입니다. 저는 이 프로젝트에서 AI를 코드 생성 도구가 아니라 문제 분해, 구현, 검증, 문서화, 운영 개선을 함께 수행하는 협업자로 사용했고, AI가 만든 결과물까지 최종 품질은 제가 책임지는 방식으로 작업했습니다.

## 3. 3분 프로젝트 설명

TownPet은 반려인이 지역과 상황별 정보를 빠르게 찾고 공유할 수 있게 만든 서비스입니다.

처음 문제는 "반려 정보가 한 채널에 뒤섞여 탐색 비용이 높다"였습니다. 병원 후기, 입양, 산책, 거래, 분실/목격은 사용자의 의도와 필요한 정보 구조가 다릅니다. 그래서 자유 게시판 하나로 시작하지 않고, `LOCAL/GLOBAL` 피드 정책과 상황별 구조화 게시판으로 문제를 나눴습니다.

백엔드에서는 단순 CRUD보다 운영 가능한 커뮤니티를 목표로 했습니다. 인증, guest write, rate-limit, 신고/제재, 관리자 감사 로그, notification outbox, production health check, 배포 preflight를 넣었고, validation, route/action, service, query, DB 책임을 나눴습니다.

AI agent는 전 과정에서 사용했습니다. 하지만 결과를 그대로 믿지 않고, 작업마다 완료 기준을 잡고 targeted test, Playwright e2e, typecheck, lint, quality gate, 문서 동기화 check, 배포 결과로 검증했습니다. 작업 이력은 `PLAN/DONE`에 남겨 세션이 바뀌어도 맥락이 이어지게 했습니다.

가장 강한 사례는 두 가지입니다. 첫째, Vercel 배포 실패가 반복됐을 때 build preflight와 운영자용 strict check의 책임을 분리해 재발을 막았습니다. 둘째, 분실/목격 기능을 단순 게시글 타입이 아니라 랜딩, 작성, 공유, 전단 저장, 목격 댓글, privacy, 모바일 e2e까지 하나의 사용자 여정으로 완성했습니다.

## 4. 대표 답변: AI 활용

질문:
> AI를 활용해 실제 생산성을 높인 경험을 설명해 주세요.

답변:

Vercel 배포가 `security env preflight failed`로 반복 실패한 적이 있습니다. 처음에는 단순 환경변수 문제처럼 보였지만, 스크립트를 따라가 보니 build 단계에서 production 필수 env 체크와 원격 `/api/health` control-plane 진단이 섞여 있었습니다. 이 구조에서는 실제 필수 secret이 있어도 원격 health 상태 때문에 unrelated 변경 후 배포가 다시 실패할 수 있었습니다.

AI agent와 함께 로그, `vercel-build.ts`, `check-security-env.ts`, package script를 추적하며 원인을 좁혔고, `check-security-env`를 `build` profile과 `full/strict` profile로 분리했습니다. 이후 Vercel build는 필수 env만 빠르게 확인하고, 원격 운영 상태 점검은 운영자용 strict check로 분리했습니다. targeted Vitest, lint, typecheck, placeholder env 재현, 실제 Vercel 재배포 성공으로 검증했습니다.

이 사례에서 AI는 원인 후보를 빠르게 좁히고 수정 범위를 탐색하는 데 도움이 됐습니다. 하지만 최종 판단은 "빌드 hot path와 운영 진단의 책임이 섞여 있다"는 구조적 문제를 제가 확인하고 분리한 것입니다. 저는 AI 활용의 핵심이 빠른 생성보다 검증 가능한 문제 해결 루프를 만드는 것이라고 생각합니다.

## 5. 대표 답변: 복잡한 요구사항 구조화

질문:
> 복잡한 요구사항을 어떻게 구조화했나요?

답변:

분실/목격 기능을 만들 때 단순히 `LOST_FOUND` 게시글 타입을 추가하는 문제로 보지 않았습니다. 실제 사용자는 급한 상황에서 등록하고, 주변 사람에게 공유하고, 목격 제보를 받아야 합니다. 그래서 `/lost-found` 랜딩, `/lost/new` 직접 등록, 구조화 분실 템플릿, 공유 도구, 전단 SVG 저장, 목격 댓글, 보호자에게만 공개할 정보의 privacy를 하나의 hot path로 설계했습니다.

검증도 단위 테스트만으로 끝내지 않았습니다. 모바일에서 실제로 랜딩 진입, 작성, 공유 패널 확인, 목격 댓글 제출, 댓글 수 동기화, private 제보 비노출까지 Playwright e2e로 고정했습니다. 다만 이 e2e는 비용이 큰 편이라 기본 hot path에는 넣지 않고 `test:e2e:lost-found` on-demand 명령으로 분리했습니다.

이 경험은 요구사항을 기능 목록이 아니라 사용자 여정, 정책, privacy, 검증 비용으로 나눠야 한다는 점을 배운 사례입니다.

## 6. 대표 답변: AI의 한계

질문:
> AI의 한계를 어떻게 다뤘나요?

답변:

AI는 빠르게 코드를 만들고 가능성을 탐색하는 데 강하지만, 현재 저장소의 운영 맥락이나 생성 파일의 side effect를 놓칠 때가 있습니다. 그래서 저는 AI가 만든 결과를 바로 신뢰하지 않고, 항상 검증 루프를 붙였습니다.

예를 들어 문서 파일을 새로 만들었을 때 문서 동기화 리포트가 stale이 되어 CI가 실패한 적이 있습니다. 원인은 `refresh-docs-index`가 `git ls-files` 기준으로 동작하는데, 새 파일을 stage하기 전에 refresh해서 로컬 check와 CI check가 다르게 나온 것이었습니다. 이후 "새 문서/스크립트/API/migration 추가 후에는 stage 이후 루트 기준 docs index check" 규칙을 추가했습니다.

AI의 한계는 "좋아 보이는 산출물"과 "실제 CI/배포/운영에서 맞는 산출물" 사이의 차이라고 생각합니다. 저는 그 차이를 테스트, 문서 freshness, 배포 로그, production health로 줄이는 쪽에 집중합니다.

## 7. 대표 답변: 협업과 문서화

질문:
> 협업 시 어떻게 설명하고 정리하나요?

답변:

저는 복잡한 기능을 코드만으로 설명하지 않으려고 합니다. TownPet에는 README, backend case study, API route inventory, generated contract report, Mermaid sequence diagrams가 있습니다. 특히 sequence diagrams는 현재 구조가 MSA가 아니라 Next.js 모놀리식 앱이라는 점을 정직하게 밝히고, `UI -> Route -> Service -> Query/DB -> External` 흐름으로 설명합니다.

문서화는 꾸미기보다 다음 사람이 빠르게 판단할 수 있는 구조가 중요하다고 봅니다. 그래서 "무엇을 했는지"보다 "왜 이 경계가 필요한지", "어떤 검증으로 완료했는지", "무엇은 의도적으로 보류했는지"를 남기려고 했습니다.

## 8. 공격 질문 방어

### Q. AI가 대부분 만든 것 아닌가요?

답변:

AI를 많이 쓴 것은 맞습니다. 하지만 문제 정의, 완료 기준, 검증 선택, 배포 판단, 문서 정합성 책임은 제가 맡았습니다. AI가 코드를 생성해도 그 코드가 제품 요구사항과 운영 기준을 만족하는지 판단하고 책임지는 일은 제 몫이라고 생각합니다.

추가로 말할 증거:
- `PLAN/DONE`으로 작업 단위를 관리했다.
- targeted tests와 e2e로 검증했다.
- Vercel 배포 실패를 실제로 원인 분석하고 재발 방지했다.

### Q. 실제 사용자가 많나요?

답변:

대규모 트래픽을 다뤘다고 말할 수는 없습니다. 대신 실제 배포된 서비스에서 production health, acquisition event smoke, e2e, 운영 루틴까지 이어지는 루프를 만들었습니다. 현재 제 강점은 대규모 운영 경험보다 문제를 운영 가능한 형태로 닫는 실행력입니다.

### Q. 왜 MSA나 Kafka가 없나요?

답변:

현재 문제 규모에서는 모놀리식이 맞다고 판단했습니다. 지금 단계에서 MSA를 흉내 내는 것보다, 모놀리식 안에서 정책과 read/write 책임을 분리하고 검증 가능한 구조를 만드는 것이 더 중요했습니다. 대신 route/service/query/DB/ops 경계를 분리했고, sequence diagram으로 메시지 흐름을 설명할 수 있게 정리했습니다.

### Q. 프로젝트가 너무 넓고 산만하지 않나요?

답변:

넓은 것은 맞습니다. 그래서 면접에서는 모든 기능을 나열하지 않고 두 사례에 집중하겠습니다. 하나는 Vercel 배포 실패 재발 방지이고, 다른 하나는 분실/목격 hot path입니다. 전자는 운영 책임과 AI 결과 검증을 보여주고, 후자는 사용자 여정과 제품 요구사항 구조화를 보여줍니다.

### Q. 가장 아쉬운 점은?

답변:

일부 backend module은 아직 큽니다. 또 production smoke는 solo 운영 기준이라 full observability는 아닙니다. 다만 이 한계를 숨기지 않고 backend case study와 DONE에 남겼고, 다음 개선 후보를 작은 단위로 관리하고 있습니다. 완벽한 프로젝트라기보다, 현재 제약과 tradeoff를 알고 계속 닫아가는 프로젝트라고 말하고 싶습니다.

## 10. 마지막 질문 답변

질문:
> 왜 우리가 뽑아야 하나요?

답변:

저는 아직 배울 것이 많지만, 문제를 흐리게 두지 않고 작게 쪼개서 실제 동작하는 결과로 닫는 방식은 꾸준히 훈련해 왔습니다. AI를 사용해 속도만 높이는 것이 아니라, AI가 만든 결과를 검증하고 운영 가능한 형태로 정리하는 데 집중했습니다. 사이오닉의 Fellowship에서 실제 고객 문제를 다루게 된다면, 모르는 영역도 빠르게 학습하되 결과의 품질과 책임을 끝까지 놓지 않는 방식으로 기여하고 싶습니다.
