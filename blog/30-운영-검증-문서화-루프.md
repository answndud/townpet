# 30. 운영 검증과 문서화 루프를 닫는 법

## 이번 글에서 풀 문제

TownPet에서 작업 완료는 코드가 올라간 순간이 아닙니다.

실제 운영 URL에서:

- 배포가 떴는지
- 핵심 화면이 깨지지 않는지
- 비로그인 사용자가 보면 안 되는 화면이 막히는지
- 운영 작업이 dry-run과 post-audit 증거를 남겼는지
- 그 결과가 문서와 report에 남았는지

까지 확인해야 완료로 봅니다.

이번 글은 성능 개선 이후 이어진 운영 검증 루프를 정리합니다.

## 왜 이 글이 필요한가

앞선 글들은 품질 게이트와 배포 파이프라인을 이렇게 나눴습니다.

- PR hot path는 작게 유지한다.
- 무거운 browser smoke는 on-demand로 보낸다.
- deploy build에는 deploy-essential만 둔다.
- production URL 검증은 post-deploy에서 따로 본다.

하지만 이 구조는 원칙만으로는 부족합니다.

실제로 한 번의 운영 작업이 끝났을 때 어떤 증거를 남겨야 하는지, 그리고 그 증거가 어디 문서에 연결되는지를 정해야 재현 가능한 운영 루프가 됩니다.

## 1. 운영 작업은 audit -> dry-run -> apply -> post-audit로 나눈다

운영 데이터나 외부 저장소를 건드리는 작업은 한 번에 실행하지 않습니다.

TownPet의 legacy upload path 정리 루프는 다음 순서로 닫았습니다.

1. 현재 production 상태를 audit한다.
2. 삭제/정리 대상을 dry-run으로 계산한다.
3. 사용자가 적용을 승인하면 apply한다.
4. 같은 기준으로 post-audit를 다시 돌린다.
5. apply report와 post-audit report를 `docs/reports/`에 남긴다.

이 방식의 장점은 단순합니다.

- 삭제 대상과 실제 삭제 결과를 분리해서 볼 수 있다.
- 운영 작업을 나중에 재검토할 수 있다.
- 실패했을 때 apply 전 문제인지 apply 후 문제인지 구분된다.

Spring/Django 기준으로 보면 migration이나 management command를 바로 실행하는 대신, `--dry-run`, `--apply`, `--verify`를 명시적으로 나누는 방식에 가깝습니다.

## 2. 화면 smoke는 public과 auth/local로 분리한다

상세 화면 visual smoke는 두 층으로 나눴습니다.

### public detail visual smoke

비로그인 사용자에게 공개 가능한 상세 화면을 봅니다.

대상 예:

- 자유게시판
- 실종 목격 제보
- 중고/공동구매
- 산책/모임

여기서는 desktop/mobile screenshot을 남기고, 화면이 렌더링되는지와 주요 텍스트가 보이는지를 확인합니다.

### auth/local detail visual smoke

로그인과 동네 맥락이 필요한 상세 화면을 봅니다.

대상 예:

- 병원 리뷰
- 돌봄 요청

이 smoke는 단순히 화면이 보이는지만 보지 않습니다.

먼저 비로그인 접근이 막히는지 확인하고, 그다음 인증/로컬 맥락에서 상세 화면이 정상 렌더링되는지 확인합니다.

즉 성공 조건은 두 가지입니다.

- guest는 접근 제한을 봐야 한다.
- auth/local 사용자는 실제 상세 화면을 봐야 한다.

이 구분이 없으면 보호되어야 할 화면이 실수로 public smoke에서만 통과할 수 있습니다.

## 3. post-deploy runner는 여러 smoke를 하나의 증거로 묶는다

최종 실행 단위는 `ops:check:detail-visual`입니다.

이 runner는 다음을 순서대로 묶습니다.

1. `ops:check:health`
2. public detail visual smoke
3. auth/local detail visual smoke
4. 통합 report 생성

대표 실행 형태는 아래와 같습니다.

```bash
OPS_BASE_URL=https://townpet.vercel.app \
AUTH_LOCAL_DETAIL_SMOKE_EMAIL=<test-user-email> \
AUTH_LOCAL_DETAIL_SMOKE_PASSWORD=<test-user-password> \
AUTH_LOCAL_DETAIL_SMOKE_CONFIRM=1 \
corepack pnpm -C app ops:check:detail-visual
```

운영 문서에서는 이 명령을 배포 전후 on-demand 체크로 연결했습니다.

중요한 점은 이 smoke가 PR hot path가 아니라는 것입니다.

화면 캡처와 로그인 맥락이 필요한 검증은 느리고 환경 의존성이 있습니다. 그래서 every push gate에 넣지 않고, release 후보나 배포 후 확인 지점에서 실행합니다.

## 4. docs-quality가 잡아준 누락

이번 루프에서 문서 품질 게이트도 실제로 역할을 했습니다.

detail visual smoke report를 추가한 뒤 report README가 ignored 경로에 걸려 누락될 수 있었습니다. 이때 `docs:refresh`와 docs-quality 체크가 문서 인덱스 불일치를 드러냈고, report README를 명시적으로 포함해 다시 닫았습니다.

여기서 얻은 교훈은 두 가지입니다.

- report는 생성만 해서는 부족하고 git에 포함되어야 한다.
- 문서 인덱스 검증은 작은 작업처럼 보여도 운영 증거 누락을 막는 역할을 한다.

## 5. PLAN / DONE은 active 상태를 닫는 문서다

TownPet의 앱 작업 상태는 블로그가 아니라 아래 문서가 기준입니다.

- [`docs/PLAN.md`](../docs/PLAN.md)
- [`docs/DONE.md`](../docs/DONE.md)

이번 루프가 끝난 뒤 active plan은 `현재 active 작업 없음`으로 닫았습니다.

블로그는 그 사실을 대체하지 않습니다.

블로그의 역할은 운영 증거와 판단 과정을 독자가 이해할 수 있게 풀어 쓰는 것입니다. 실제 작업 상태의 source of truth는 계속 `docs/`에 둡니다.

## 6. 면접에서 이렇게 설명할 수 있다

> TownPet에서는 배포가 끝났다는 기준을 build green으로 보지 않았습니다. production URL health, public detail visual smoke, auth/local detail smoke, 운영 cleanup dry-run/apply/post-audit report, docs-quality까지 묶어서 증거를 남겼습니다. 특히 보호되어야 하는 병원 리뷰나 돌봄 요청 상세는 guest 접근 제한과 인증 후 렌더링을 둘 다 확인했습니다.

이 답변은 단순히 “테스트를 많이 했다”가 아닙니다.

핵심은 검증 위치를 나눴다는 점입니다.

- 빠른 회귀는 PR hot path
- 무거운 브라우저 확인은 on-demand
- 실제 운영 URL 확인은 post-deploy
- 운영 작업 증거는 report
- 현재 작업 상태는 `PLAN/DONE`

이렇게 나누면 검증은 유지하면서도 개발 루프를 불필요하게 느리게 만들지 않습니다.

## 산출물 체크리스트

이번 운영 검증 루프가 닫혔다고 말하려면 아래가 있어야 합니다.

- production cleanup dry-run/apply/post-audit report
- public detail visual smoke screenshot report
- auth/local detail visual smoke screenshot report
- 통합 detail visual smoke report
- 운영 문서의 실행 명령과 확인 지점
- docs-quality 통과
- `PLAN/DONE` active 상태 정리

이 중 하나라도 빠지면 작업은 “코드는 됐지만 운영 증거가 부족한 상태”로 남습니다.

## 현재 구현의 한계

- auth/local visual smoke는 운영 테스트 계정과 환경 변수가 필요합니다.
- screenshot 검증은 layout collapse나 텍스트 누락을 빠르게 잡지만, 모든 접근성 문제를 자동으로 보장하지는 않습니다.
- report가 많아질수록 오래된 증거와 최신 증거를 구분하는 naming 규칙이 더 중요해집니다.

## Python/Java 개발자용 요약

- `ops:check:detail-visual`은 Spring 프로젝트의 post-deploy smoke runner에 가깝습니다.
- public smoke와 auth smoke를 분리한 것은 controller 권한 테스트와 실제 브라우저 렌더링 확인을 함께 보는 구조입니다.
- `docs/reports/`는 테스트 로그가 아니라 운영 증거 archive입니다.
- `docs-quality`는 README나 운영 문서의 링크/인덱스가 현재 산출물과 맞는지 확인하는 lightweight consistency gate입니다.
