# 31. 성능 개선 8단계를 수치로 닫은 기록

## 왜 다시 성능을 봤는가

TownPet 배포 페이지는 첫 진입과 게시판 이동이 느리게 느껴졌다. 프레임워크나 DB를 바꾸기 전에, 먼저 “무엇이 느린지”를 수치로 나눴다.

바꾸지 않은 것:

- Next.js 유지
- PostgreSQL 유지
- Vercel 배포 구조 유지

대신 측정과 hot path를 고쳤다.

## Phase 1. 기준선 고정

production baseline에서 확인한 값:

| 항목 | 값 |
| --- | ---: |
| warm `/` server response | `108ms` |
| warm `/feed/guest` server response | `106ms` |
| warm `/api/home/feed` | `105ms` |
| warm `/api/feed/guest?limit=20` | `105ms` |
| cold `/` first request | `1145ms` |
| cold public detail first request | `1826ms` |

결론은 warm 응답보다 cold outlier, client fetch, RSC prefetch, route asset이 문제라는 것이었다.

## Phase 2. Web Vitals 수집 루프

실사용 데이터를 쌓기 위해 `WebVitalSample`을 추가했다.

수집 metric:

- LCP
- INP
- CLS
- FCP
- TTFB

개인정보를 피하기 위해 route는 aggregate key로 저장한다.

```text
/posts/cmp123/guest -> /posts/[id]/guest
/users/user-1       -> /users/[id]
/media/uploads/a    -> /media/[path]
```

운영 조회는 `pnpm -C app perf:web-vitals`로 p75/p95 summary를 만든다.

## Phase 3. 홈 live board를 server-rendered ISR로 전환

기존 홈은 정적 shell 뒤에 client component가 `/api/home/feed`를 호출했고, 그 전까지 skeleton이 보였다.

변경:

- 홈 피드 직렬화 로직을 `getHomeFeedPayload()` query로 분리
- `/api/home/feed`와 `/`가 같은 query 사용
- `HomeFeedPreview`에서 client fetch 제거
- `/`는 `revalidate = 60` ISR 페이지로 변경

확인:

- 첫 HTML에 `먼저 확인할 글`, `최근 올라온 공개 글`, empty action 포함
- `animate-pulse` skeleton 없음
- `/api/home/feed` client fetch 없음

## Phase 4. Link prefetch 폭주 제거

Playwright로 mobile 첫 진입 resource를 보니, 화면에 보이는 링크들이 클릭 전부터 RSC fetch를 만들고 있었다.

before:

| route | fetch count | fetch transfer | total transfer |
| --- | ---: | ---: | ---: |
| `/` | `34` | `32KB` | `288KB` |
| `/feed/guest` | `51` | `84KB` | `433KB` |

변경:

- dense link list는 `prefetch={false}`
- feed row/detail thumbnail/author/ad CTA prefetch 제거
- guest feed mount 직후 post detail `router.prefetch()` 제거
- public acquisition header에서는 viewer shell fetch skip

after:

| route | fetch count | fetch transfer | total transfer |
| --- | ---: | ---: | ---: |
| `/` | `0` | `0KB` | `242KB` |
| `/feed/guest` | `3` | `17KB` | `273KB` |

## Phase 5. API outlier를 phase로 분리

`perf=1`을 붙이면 `Server-Timing`으로 내부 phase를 볼 수 있게 했다.

local production snapshot:

| route | first | p50 | p95 | 주요 phase |
| --- | ---: | ---: | ---: | --- |
| `/api/health?perf=1` | `134.3ms` | `13.5ms` | `134.3ms` | `health_snapshot` |
| `/api/home/feed?perf=1` | `35.5ms` | `5.6ms` | `35.5ms` | `home_feed_query` |
| `/api/feed/guest?perf=1` | `18.3ms` | `9.3ms` | `18.3ms` | `bootstrap.policy_and_communities` |

## Phase 6. guest feed 기본 path query 줄이기

`/api/feed/guest` 기본 요청은 petType catalog가 없어도 된다. 그런데 기존에는 기본 요청에서도 `listCommunityNavItems(50)`가 bootstrap에 포함됐다.

변경:

- 기본 요청: `bootstrap.policy`만 실행
- petType query가 있을 때만 `bootstrap.communities` 실행

phase total p95는 `9.0ms -> 6.0ms`로 줄었다.

## Phase 7. webfont 제거

Next font로 Space Grotesk, IBM Plex Mono latin font가 생성되고 있었다. TownPet은 한국어 UI가 대부분이라 system font stack이 더 적절했다.

변경:

- `next/font/google` 제거
- `--font-sans`를 OS 한글/system stack으로 변경
- `--font-mono`는 system monospace stack 사용

after build:

```text
.next/static/media
└── favicon.0b3bf435.ico
```

font 파일이 사라졌다.

## 추가 하드닝: public detail 90점 이상 목표

Phase 8 직후에도 public detail은 다른 핫패스보다 client script가 컸다.

Phase 8 직후 public detail mobile asset:

| 항목 | 값 |
| --- | ---: |
| total transfer | `306KB` |
| script transfer | `257KB` |
| CSS transfer | `37KB` |
| fetch transfer | `11KB` |
| mobile LCP p50 | `664ms` |
| mobile LCP p95 | `872ms` |

나쁘지는 않았지만 `/`, `/login`, `/feed/guest`와 비교하면 상세 페이지의 client script가 컸다. 원인은 게시글 상세가 댓글, 반응, 북마크, 공유, 신고, 비회원 수정/삭제, 메뉴/팝오버, 이미지 표시 같은 상호작용을 한 화면에 많이 들고 있었기 때문이다.

추가로 적용한 방식은 기능 제거가 아니라 초기 hydrate 범위 축소였다.

- 댓글 섹션은 placeholder만 먼저 렌더링하고, hash 이동/근접 scroll/`댓글 열기` 클릭 시 로드한다.
- 게스트 상세의 좋아요/싫어요/북마크/공유는 로그인 사용자용 mutation 컴포넌트 대신 guest 전용 action bar로 분리했다.
- 비회원 글 수정/삭제 도구는 `...` 메뉴를 열었을 때만 `import()`한다.
- 분실/목격 공유 패널은 분실글에서 공유 도구를 열 때만 무거운 공유/이미지 로직을 가져온다.
- 공개 상세의 목록 이동은 client router 버튼이 아니라 서버 렌더 링크로 바꿨다.
- 팝오버/메뉴 닫힘은 `pointerdown`에 더해 `mousedown`, `touchstart`, `focusin`, `Escape`로 닫히게 보강했다.

production 반영 후 public detail mobile route asset:

| 항목 | Phase 8 직후 | 추가 하드닝 후 | 개선 |
| --- | ---: | ---: | ---: |
| total transfer | `306KB` | `226KB` | `-80KB`, `26.1%` 감소 |
| script transfer | `257KB` | `183KB` | `-74KB`, `28.8%` 감소 |
| CSS transfer | `37KB` | `30KB` | `-7KB`, `18.9%` 감소 |
| fetch transfer | `11KB` | `13KB` | `+2KB` |

production browser 측정:

| route | mobile LCP p50 | mobile LCP p95 | total goto p50 |
| --- | ---: | ---: | ---: |
| `/` | `204ms` | `608ms` | `220ms` |
| `/login` | `328ms` | `616ms` | `358ms` |
| `/feed/guest` | `192ms` | `196ms` | `223ms` |
| public detail | `668ms` | `1164ms` | `497ms` |

public detail의 asset budget은 목표였던 total `<= 280KB`, script `<= 220KB`를 통과했다. LCP p50은 기존 `664ms`와 거의 같은 `668ms`라 체감 paint는 유지했고, 초기 JS 전송량은 크게 줄었다. p95는 네트워크와 서버 응답 편차가 섞여 있어 production 표본을 더 쌓아 봐야 한다.

핵심은 프레임워크 교체가 아니었다.

- 첫 HTML에 필요한 데이터를 넣기
- 클릭 전 prefetch 폭주 막기
- API phase를 나눠 보기
- 불필요한 webfont 제거하기

이 네 가지가 가장 큰 효과를 냈다.

## 운영 budget

운영 기준은 [성능_budget.md](../business/operations/성능_budget.md)에 고정했다.

핫패스 budget:

- `/`: mobile total transfer `<= 250KB`
- `/login`: mobile total transfer `<= 280KB`
- `/feed/guest`: mobile total transfer `<= 300KB`
- `/api/feed/guest?perf=1`: local p95 `<= 100ms`

이 budget은 매 배포마다 강제하지 않는다. 1인 운영 기준에서는 성능 관련 코드를 건드렸을 때만 on-demand로 확인한다.
