# 29. 성능 개선을 측정하고 병목을 분리한 기록

## 이번 글에서 풀 문제

TownPet production 페이지를 직접 써보면 첫 진입, 게시판 이동, 댓글 작성이 느리게 느껴질 수 있다는 피드백이 있었습니다.

처음 떠올릴 수 있는 해결책은 큽니다.

- Next.js를 버리고 Vite + React로 바꾸기
- PostgreSQL을 버리고 SQLite로 바꾸기
- 서버 렌더링을 줄이고 전부 클라이언트 앱처럼 만들기

하지만 이번 작업에서는 프레임워크와 DB를 바꾸지 않았습니다.

대신 먼저 질문을 나눴습니다.

1. 서버 HTML 응답이 느린가?
2. 브라우저가 JS를 받고 hydrate하는 비용이 큰가?
3. DB query가 느린가?
4. mutation은 실제로 느린가, 아니면 화면 반응이 늦게 보이는가?
5. cold start와 region 문제가 섞여 있는가?

결론부터 말하면, 짧은 production 샘플에서는 “항상 10초 걸리는 서버 TTFB”는 재현되지 않았습니다. 그래서 이번 개선은 큰 교체가 아니라 **측정 축을 분리하고, 확인된 낭비를 줄이는 방향**으로 진행했습니다.

## 1. 먼저 “느리다”를 쪼갰다

사용자가 말하는 “느림”은 하나의 숫자가 아닙니다.

TownPet에서는 다음 네 가지를 따로 봤습니다.

| 축 | 질문 | 측정 도구 |
| --- | --- | --- |
| 서버 응답 | HTML/API 첫 byte와 다운로드가 느린가? | `measure-production-performance.ts` |
| 브라우저 paint | FCP/LCP가 늦는가? | `measure-browser-performance.ts` |
| route asset | JS/CSS/fetch transfer가 큰가? | `measure-route-assets.ts` |
| DB readiness | region/pooler/index가 병목 후보인가? | `measure-db-readiness.ts` |

이렇게 나눈 이유는 단순합니다.

서버 응답이 200ms인데 화면이 2초 늦게 보이면 DB를 바꿀 문제가 아닙니다. 반대로 브라우저 리소스가 가벼운데 API가 매번 1초를 넘으면 bundle을 줄여도 효과가 작습니다.

## 2. baseline에서 확인한 것

production Node fetch 기준 짧은 샘플은 아래와 같았습니다.

| target | first total |
| --- | ---: |
| `/` | `333ms` |
| `/feed` | `311ms` |
| `/feed/guest` | `120ms` |
| `/api/health` | `355ms` |

Playwright desktop 기준 production paint는 아래와 같았습니다.

| target | FCP | LCP |
| --- | ---: | ---: |
| `/` | `632ms` | `632ms` |
| `/feed` | `424ms` | `424ms` |
| `/feed/guest` | `236ms` | `248ms` |

여기서 중요한 판단은 이겁니다.

**짧은 production 샘플만 보면 서버 HTML 응답이 항상 10초 걸리는 상태는 아니었습니다.**

그래서 “Next.js가 느리다”, “PostgreSQL이 느리다”로 결론내지 않고, 화면 구조와 query 경로를 줄이는 작업부터 했습니다.

## 3. `/`를 빠른 shell로 바꿨다

기존 홈은 `/feed`로 보내는 redirect 중심이었습니다.

문제는 첫 방문자가 제품 설명을 보기 전에 feed 비용을 먼저 기다린다는 점입니다. 게다가 root layout에서 header 커뮤니티 메뉴를 위해 서버가 커뮤니티 목록을 기다리면, 홈을 정적으로 만들더라도 첫 HTML이 DB 상태에 영향을 받을 수 있습니다.

바꾼 구조는 이렇습니다.

```text
/                 static product shell
/api/home/feed    delayed popular/latest preview
/api/communities  client-side delayed header data
```

핵심은 `/`가 먼저 뜨고, 피드 preview는 나중에 채우는 것입니다.

로컬 개발 서버 warm 측정에서는 `/` FCP/LCP가 `124ms/124ms`까지 내려갔습니다. 같은 조건에서 `/feed`는 `908ms/908ms`, `/feed/guest`는 `416ms/416ms`였습니다.

여기서 얻은 결론은 명확합니다.

홈 첫 화면은 피드 전체 read model을 기다리게 만들 필요가 없습니다. 첫 화면의 역할은 “서비스가 무엇인지 빠르게 전달하고, 다음 행동으로 보내는 것”입니다.

## 4. feed 첫 페이지의 count query를 줄였다

피드 첫 페이지는 거의 항상 `page=1`입니다.

그런데 첫 페이지에서 전체 count를 먼저 세고 list를 다시 기다리면, 실제 첫 화면에는 필요 없는 비용을 매번 냅니다.

그래서 `resolveFeedPageSlice`에 `skipCountOnFirstPage`를 넣었습니다.

규칙은 단순합니다.

1. 첫 페이지는 `listPage(1)`만 실행합니다.
2. `limit + 1` 방식의 `nextCursor`가 있으면 다음 페이지가 있다고 봅니다.
3. 2페이지 이상이나 overflow 보정이 필요한 경우에는 기존처럼 count를 사용합니다.

이 선택은 정확성을 일부 포기한 것이 아닙니다.

첫 페이지에서 “정확한 전체 페이지 수”보다 “첫 카드 목록을 빨리 보여주는 것”을 우선하고, 정확한 count가 필요한 페이지 이동에서는 기존 로직을 유지한 것입니다.

## 5. 기본 feed에서 개인화 context를 늦췄다

로그인 사용자의 기본 `/feed`에서도 개인화 context를 미리 계산하면 일반 최신순/좋아요순/댓글순 클릭이 불필요한 read path에 묶입니다.

그래서 개인화 context 조회 조건을 실제 `personalized=1` 상태로 좁혔습니다.

이제 일반 feed에서는 아래 조회를 건너뜁니다.

- audience segment
- pet profile summary
- recent engagement
- recent behavior
- dwell summary
- bookmark summary

개인화 기능을 없앤 것이 아니라, 사용자가 명시적으로 개인화를 요청했을 때만 비용을 냅니다.

## 6. 댓글 작성은 실제 latency와 체감 latency를 분리했다

댓글 작성은 서버 action, DB write, count update, notification side effect가 엮일 수 있습니다.

하지만 사용자가 느끼는 첫 문제는 “눌렀는데 화면이 아무 반응이 없다”입니다.

그래서 두 가지를 바꿨습니다.

1. 서버 완료 전에도 `pending` 댓글 preview를 즉시 보여줍니다.
2. `onCommentsChanged`로 댓글 목록을 이미 다시 가져오는 화면에서는 추가 `router.refresh()`를 하지 않습니다.

이렇게 하면 서버 완료 시간이 그대로여도 사용자는 클릭 직후 상태 변화를 봅니다. 실패하면 pending preview를 제거하고 기존 오류 메시지를 보여줍니다.

이건 optimistic update와 비슷하지만, 아직 서버에서 받은 id가 없으므로 “완료된 댓글처럼 확정 표시”하지 않고 “등록 중”으로 표시했습니다. 작은 차이지만 커뮤니티 UI에서는 중요합니다.

## 7. public route의 auth-only chunk를 분리했다

전역 header는 모든 public route에 들어갑니다.

그런데 header에서 인증 사용자에게만 필요한 컴포넌트를 정적으로 import하면, 게스트 첫 화면에도 관련 코드가 initial chunk에 섞일 수 있습니다.

분리한 컴포넌트는 두 개입니다.

- `AuthControls`
- `NotificationBell`

둘 다 `next/dynamic`과 `ssr: false`로 바꿨습니다.

게스트 초기 화면은 로그인 링크만 있으면 됩니다. 로그아웃 버튼과 알림 dropdown은 viewer shell이 인증 상태를 확인한 뒤 필요할 때만 로드하면 충분합니다.

로컬 production build 기준 route asset snapshot은 아래처럼 남았습니다.

| route | script transfer | CSS transfer | FCP/LCP |
| --- | ---: | ---: | ---: |
| `/` desktop | `173KB` | `79KB` | `164/488ms` |
| `/feed/guest` desktop | `173KB` | `79KB` | `56/56ms` |
| `/feed/guest` mobile | `173KB` | `79KB` | `40/40ms` |

이 숫자는 절대값보다 “앞으로 비교할 기준”으로 중요합니다.

## 8. DB는 바꾸지 않고 readiness를 먼저 봤다

PostgreSQL을 SQLite로 바꾸는 것은 이번 문제의 해법이 아닙니다.

운영 병목을 보려면 먼저 아래를 봐야 합니다.

- Vercel runtime region
- DB provider region
- pooler 사용 여부
- query plan
- index coverage

그래서 `measure-db-readiness.ts`를 추가했습니다.

이 스크립트는 credential을 출력하지 않고 아래만 기록합니다.

- 현재 process의 DB host/pooler signal
- production header의 `x-vercel-id`
- query surface별 index signal coverage
- 다음에 실행할 `EXPLAIN (ANALYZE, BUFFERS)` 후보

production header에서는 `/`와 `/feed/guest`가 `icn1` edge signal을 보였고, `/api/feed/guest`, `/api/health`는 `icn1 -> sin1` 형태의 runtime path signal을 보였습니다.

정적 coverage 기준으로는 주요 surface에 index signal이 있었습니다.

| surface | coverage |
| --- | --- |
| feed latest | `Post_scope_status_createdAt_idx` |
| feed best | `Post_scope_status_best_order_idx` |
| local board/type | `neighborhoodId,type,status,createdAt` |
| ranked search | title/content/structured search trigram/tsvector indexes |
| comments | `postId,createdAt` |
| reports | `Report_status_targetType_createdAt_idx` |

단, 이것은 “실제 query plan이 좋다”는 뜻은 아닙니다. 데이터 분포와 파라미터에 따라 planner가 다른 선택을 할 수 있습니다.

그래서 migration은 바로 추가하지 않았습니다. 다음 단계는 production-like data에서 `EXPLAIN (ANALYZE, BUFFERS)`를 보는 것입니다.

## 9. 이번 개선에서 일부러 하지 않은 것

하지 않은 것들이 더 중요할 때가 있습니다.

이번에는 아래를 하지 않았습니다.

- Next.js 제거
- PostgreSQL 제거
- 모든 페이지 client-only 전환
- 근거 없는 index 추가
- 댓글 notification outbox 도입
- production DB credential 없는 상태에서 EXPLAIN 결과를 상상해서 쓰기

이유는 같습니다.

성능 문제는 큰 기술 교체보다, **어느 계층이 느린지 먼저 분리하는 일**이 우선이기 때문입니다.

## 10. 남은 작업

아직 끝난 것은 아닙니다.

남은 확인은 아래입니다.

1. 최신 `main`이 production에 배포된 뒤 같은 스크립트로 다시 측정합니다.
2. production DB provider region과 pooler 사용 여부를 dashboard에서 확인합니다.
3. `ranked_search`, `feed_best_global`, `feed_latest_global`, `comments_detail`, `report_queue`에 대해 `EXPLAIN (ANALYZE, BUFFERS)`를 실행합니다.
4. 실제 post detail URL을 넣어 browser/asset 측정을 확장합니다.
5. 측정 결과가 나쁘게 나온 query만 migration 또는 query rewrite 후보로 분리합니다.

## 정리

이번 작업의 핵심은 “속도를 빠르게 했다”보다 “속도를 말할 수 있게 만들었다”입니다.

TownPet는 이제 다음 숫자를 따로 볼 수 있습니다.

- 서버 total/header time
- 브라우저 FCP/LCP
- route별 JS/CSS/fetch transfer
- DB region/pooler/index readiness
- 댓글 mutation perceived latency

프레임워크와 DB를 바꾸지 않고도 할 수 있는 일은 꽤 많았습니다.

가장 큰 교훈은 이겁니다.

**성능 개선은 교체가 아니라 분리에서 시작한다.**

서버, 브라우저, DB, 사용자 체감을 분리해서 보면, 어떤 문제는 코드 한 줄보다 측정 기준 하나가 더 큰 개선을 만듭니다.
